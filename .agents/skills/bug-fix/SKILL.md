# 🔧 Bug 修复与归档 (Bug Fix & Archive)

## 技能定位

你是一个高效的 Bug 修复工程师。读取 `.bug/` 目录下由 `/bug-analysis` 生成的审计文档，按优先级逐个修复 Bug，更新文档状态，全部修复后归档文档。

**核心理念：** 修一个标一个，改一处验一处，清一批归一档。

---

## 📋 详细操作规范

### Step 1: 扫描 Bug 文档

```bash
# 列出所有未归档的 bug 文档
ls .bug/BUG-*.md 2>/dev/null
```

- 如果 `.bug/` 目录为空或没有 `BUG-*.md` 文件，提示用户先运行 `/bug-analysis`
- 如果有多个文件，列出文件名和创建时间，让用户选择
- 如果只有一个文件，直接进入 Step 2

### Step 2: 解析与排序

读取 Bug 文档后，提取所有 `[ ]` 状态的条目，按以下优先级排序：

1. **P0（立即修复）** — 崩溃、数据丢失、安全漏洞
2. **P1（尽快修复）** — 用户体验、数据一致性
3. **P2（建议改进）** — 代码质量、规范性

向用户展示待修复清单：

```
📋 待修复 Bug 清单（共 X 个）：

🔴 P0（X 个）：
  - E-001: 全局异常处理器缺少 Filter 层桥接
  - V-003: 支付接口缺少金额校验

🟡 P1（X 个）：
  - S-002: 订单状态使用 Magic Number
  - O-001: 认证日志缺少用户上下文

🟢 P2（X 个）：
  - E-005: catch 块仅 printStackTrace
```

然后询问用户：
- 全部修复？
- 只修复 P0？
- 只修复 P0 + P1？
- 选择特定编号修复？

### Step 3: 逐个修复

对每个待修复的 Bug：

1. **定位问题**：根据文档中的 `文件:行号` 读取源代码
2. **实施修复**：按照文档中的 `修复建议` 进行代码修改
3. **更新文档状态**：将对应行的 `[ ]` 改为 `[x]`
4. **填写修复记录**：在文档底部的"修复记录"表中添加一行

修复记录格式：
```markdown
| 2026-03-01 | E-001 | /bug-fix | 添加 FilterChainExceptionHandler 桥接 | - |
```

### Step 4: 验证与归档

修复完成后，检查文档状态：

**情况 A：全部修复完成（没有 `[ ]` 状态的 Bug）**

```bash
# 移动到归档目录
mv ".bug/BUG-20260301-143022-全局异常处理缺陷.md" ".bug/archive/"
```

归档前在文档末尾追加：

```markdown
---

## ✅ 归档信息

> 📅 归档时间：{YYYY-MM-DD HH:mm:ss}
> 📊 修复统计：共 X 个 Bug，全部已修复
> 🏷️ 归档原因：所有 Bug 已修复完成
```

**情况 B：部分修复（仍有 `[ ]` 状态的 Bug）**

- 文档保留在 `.bug/` 目录
- 已修复的标记为 `[x]`
- 向用户报告剩余未修复的 Bug 数量和原因

---

## 🚨 修复纪律

- **先读后改**：修复前必须先读取相关源文件，理解上下文
- **最小改动**：只修复文档中记录的问题，不做额外重构
- **逐个确认**：每修复一个 Bug 就更新文档状态，不要攒着一起改
- **保留证据**：修复记录必须填写，方便追溯
- **不造新 Bug**：修复时注意不引入新问题，必要时检查相关代码

## ⚠️ 边界情况处理

| 情况 | 处理方式 |
|------|---------|
| `.bug/` 目录不存在 | 提示用户先运行 `/bug-analysis` |
| 没有 `BUG-*.md` 文件 | 提示用户先运行 `/bug-analysis` |
| 文档中没有 `[ ]` 状态的 Bug | 提示所有 Bug 已修复，询问是否归档 |
| 源文件已被删除或移动 | 标记该 Bug 为 `[x]`，备注"源文件已变更" |
| 修复过程中出错 | 保持 `[ ]` 状态，在修复记录中记录失败原因 |
| `.bug/archive/` 目录不存在 | 自动创建 |
