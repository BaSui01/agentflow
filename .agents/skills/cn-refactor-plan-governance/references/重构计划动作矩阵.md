# 重构计划动作矩阵

## 1. 先判定读哪些文档

### A. 用户要“重构建议 / cleanup / refactor / deslop”

先读：

1. `AGENTS.md`
2. `docs/重构计划/` 中相关活动计划
3. `docs/architecture/startup-composition.md`
4. 相关 ADR

必要时再读：

5. `docs/重构计划/归档/` 中的历史计划
6. `architecture_guard_test.go`
7. `scripts/arch_guard.ps1`

优先模板：

- 如果任务明显属于“包合并 / 唯一入口 / 根目录治理 / 删除旧实现”，优先使用 `references/架构重构计划模板.md`
- 普通功能重构或范围较小的 cleanup，再使用 `references/重构计划模板.md`

### B. 用户要“推进计划 / 验收 / 收尾”

先读：

1. 目标计划
2. `scripts/refactor_plan_guard.py`
3. 相关守卫/测试声明

### C. 用户要“归档计划”

先读：

1. 目标计划
2. `python .codex/skills/cn-refactor-plan-governance/scripts/audit_plan.py "<计划文件名或路径>"` 的结果
3. `python scripts/refactor_plan_guard.py report` 的当前结果
4. 如有必要，读 `docs/重构计划/归档/README.md`

## 2. 架构对齐检查

给建议前，至少回答下面问题：

1. 是否符合单轨替换，而不是双实现并存？
2. 是否符合分层依赖方向？
3. 是否符合启动主链：`cmd -> bootstrap -> routes -> handlers -> domain`？
4. `api/handlers` 是否只做协议适配？
5. 是否应收口到 `Builder` / `Factory` / `Registry`，而不是继续增加散落构造？
6. 是否需要同步更新 README / ADR / 架构文档 / 守卫？
7. 是否已经给出“职责矩阵”和“重复语义 -> 唯一归口”？
8. 是否写清楚“保留 / 合并 / 下沉 / 删除”的具体对象？
9. 是否涉及根目录治理、目录预算或 allowlist？

## 3. 建议输出模板

### A. 重构建议

- 结论
- 读取的计划 / 架构文档
- 当前冲突点
- 职责矩阵（保留 / 合并 / 下沉 / 删除）
- 重复语义与唯一归口
- 建议动作（具体到文件/目录/入口）
- 删除清单（旧入口 / 旧文件 / 旧路径 / 旧守卫）
- 文档与守卫同步项
- 验证命令

### B. 计划推进

- 完成了什么
- 哪些 `[ ]` 改为 `[x]`
- 证据是什么
- 剩余多少 `[ ]`
- 下一步是什么
- 若是架构类计划：补充本轮已合并/已删除/已下沉的对象清单

### C. 计划归档

- 目标计划是否 `todo = 0`
- 是否仍是当前执行基线
- 已修正哪些引用
- `report` / `gate` 结果如何
- `audit_plan.py` 的 verdict 是什么
- 明确区分：
  - “该计划可归档”
  - “仓库整体可收尾”

## 4. 命令建议

```powershell
python scripts/refactor_plan_guard.py lint --target "<文件名>" --require-tdd --require-verifiable-completion
python scripts/refactor_plan_guard.py report --target "<文件名>" --require-tdd --require-verifiable-completion
python scripts/refactor_plan_guard.py gate --target "<文件名>" --require-tdd --require-verifiable-completion
python .agents/skills/cn-refactor-plan-governance/scripts/run_guard.py lint --target "<文件名>"
python .agents/skills/cn-refactor-plan-governance/scripts/run_guard.py report --target "<文件名>"
python .agents/skills/cn-refactor-plan-governance/scripts/run_guard.py gate --target "<文件名>"
python .agents/skills/cn-refactor-plan-governance/scripts/audit_plan.py "<计划文件名或路径>"
python .agents/skills/cn-refactor-plan-governance/scripts/audit_plan.py --target all
python .agents/skills/cn-refactor-plan-governance/scripts/audit_plan.py --target all --json --fail-on not-ready
python .agents/skills/cn-refactor-plan-governance/scripts/audit_plan.py "<计划文件名或路径>" --json --fail-on direct-references
```

## 5. 常见误判

- 把归档计划当作当前基线
- 因为某个计划完成，就宣布整个仓库已收尾
- 没有证据就把 `[ ]` 改成 `[x]`
- 只写“建议优化架构”，却不给具体入口、删除项、验证项
- 只写“拆包/合并”却没有说明职责矩阵与唯一归口
- 只移动文件，不删除旧实现或旧文档口径
- 忽略 `AGENTS.md` 中的层级约束、启动链路和“禁止兼容代码”
