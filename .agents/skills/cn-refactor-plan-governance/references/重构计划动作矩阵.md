# 重构计划动作矩阵

## 1. 模式路由

### A. 诊断模式

用户："分析代码/扫描/体检/诊断/生成计划/有什么问题/帮我看看"

**读取清单**：
1. `CLAUDE.md`（分层规则、依赖方向、启动主链）
2. 目标范围目录结构
3. `architecture_guard_test.go`（现有守卫）
4. `references/诊断命令参考.md`（扫描命令）

**输出模板**：诊断报告
**模板选择**：架构类→`架构重构计划模板.md`，普通→`重构计划模板.md`

### B. 执行模式

用户："执行/推进/继续/按计划改/做任务"

**读取清单**：
1. `docs/重构计划/` 下所有非归档计划
2. 未完成任务涉及的所有源代码文件
3. `CLAUDE.md`（架构约束）

**优先级**：Phase 靠前 > 阻塞任务 > 影响范围小 > 有验证命令

**输出模板**：执行报告

### C. 治理模式

用户："审核/检查/lint/report/gate/归档/收尾"

**读取清单**：
1. 目标计划
2. `scripts/refactor_plan_guard.py`
3. `audit_plan.py`

**输出模板**：推进报告 / 归档报告

## 2. 架构对齐检查（通用）

1. 是否符合单轨替换？
2. 是否符合分层依赖方向？
3. 是否符合启动主链：`cmd → bootstrap → routes → handlers → domain`？
4. `api/handlers` 是否只做协议适配？
5. 是否应收口到 `Builder/Factory/Registry`？
6. 是否写清"保留/合并/下沉/删除"的具体对象？
7. 是否涉及根目录治理？
8. 是否需要同步 README/ADR/架构文档/守卫？

## 3. 输出模板

### 诊断报告

```markdown
## 诊断报告（YYYY-MM-DD）

### 扫描范围
- N 个包，M 个文件

### 发现问题
| 级别 | 数量 | 典型问题 |
|------|------|----------|
| P0 | X | ... |
| P1 | X | ... |
| P2 | X | ... |
| P3 | X | ... |

### 生成计划
- 文件：`docs/重构计划/xxx.md`
- Phase：N，任务：M

### 建议
1. ...
```

### 执行报告

```markdown
## 执行报告（YYYY-MM-DD）

### 本轮完成任务
1. Phase-X 任务 → `[x]`

### 修改文件
| 文件 | 内容 |
|------|------|
| `a.go` | ... |

### 验证结果
| 命令 | 结果 |
|------|------|
| `go test ./pkg` | 通过 |

### 剩余未完成（N 项）
1. ...（阻塞原因）

### 阻塞/风险
- ...

### 下一步
- ...
```

### 治理：推进/归档

- 完成项：从 `[ ]` 改 `[x]`+ 证据
- 剩余 `[ ]` 数量
- gate 结果
- audit_plan 的 verdict
- 明确："可归档" vs "可收尾"

## 4. 命令速查

| 场景 | 命令 |
|------|------|
| 格式检查 | `python scripts/refactor_plan_guard.py lint --target "<文件>" --require-tdd --require-verifiable-completion` |
| 进度报告 | `python scripts/refactor_plan_guard.py report --target "<文件>" --require-tdd --require-verifiable-completion` |
| 门禁检查 | `python scripts/refactor_plan_guard.py gate --target "<文件>" --require-tdd --require-verifiable-completion` |
| 归档审计 | `python .agents/skills/cn-refactor-plan-governance/scripts/audit_plan.py --target all` |
| 诊断扫描 | 见 `references/诊断命令参考.md` |
| 验证：单包 | `go test ./target/pkg -count=1` |
| 验证：单测 | `go test . -run "TestName" -count=1` |
| 验证：架构 | `powershell -ExecutionPolicy Bypass -File scripts/arch_guard.ps1` |

## 5. 常见误判

### 诊断
- 只看目录不看代码 → 每个问题必须有文件+行号
- 定级随意 → P0=硬规则违反，P1=明确技术债
- 生成计划后不跟用户确认 → 必须等用户说"可以"

### 执行
- 只跑 lint 不改代码 → 那不是执行
- 遇到第一个任务就改，不做全局排序 → 先排优先级
- 改完不验证就勾 `[x]` → 验证命令不过不算完成
- 改完代码不更新计划 → 同步更新是强制要求
- 顺手做计划外的"小优化" → 严格按计划范围
- 验证失败修无关已有问题 → 只修本轮引入的

### 治理
- 归档计划当基线
- 无证据就改 `[x]`
- 只写抽象建议不给具体路径
- 移动文件不删除旧实现
- 忽略 CLAUDE.md 的架构约束
