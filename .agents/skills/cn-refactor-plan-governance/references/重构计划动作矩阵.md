# 重构计划动作矩阵

## 1. 模式路由

### A. 诊断模式

用户说："分析/扫描/体检/诊断/有什么问题/帮我看看代码/生成计划"

**读取清单**：
1. `CLAUDE.md`（分层规则、启动主链、架构约束）
2. 目标范围目录结构
3. `architecture_guard_test.go`（现有守卫规则）
4. `references/诊断命令参考.md`（扫描命令集）

**执行**：定范围 → 五维扫描 → 定级 P0-P3 → 生成计划 → 确认 → 输出报告

**模板**：架构类→`架构重构计划模板.md`，普通→`重构计划模板.md`

### B. 执行模式

用户说："执行/推进/继续/按计划改/做任务/完成剩余"

**读取清单**：
1. `docs/重构计划/` 下所有活跃计划（排除 `归档/`）
2. 未完成任务涉及的所有源文件
3. `CLAUDE.md`

**优先级**：Phase 靠前 > 阻塞任务 > 影响小 > 有验证命令

**执行**：扫描汇总 → 排优先级 → 逐项读代码 → 修改 → 验证 → 更新计划 → 输出报告

### C. 治理模式

用户说："审核/检查/lint/report/gate/归档/收尾/验收"

**读取清单**：
1. 目标计划文件
2. `scripts/refactor_plan_guard.py`
3. `scripts/audit_plan.py`

**执行**：跑 lint/report/gate → 判断是否可归档/可收尾

## 2. 架构对齐检查（通用）

1. 是否符合单轨替换？不禁用兼容双实现
2. 是否符合分层依赖方向：`types → llm → agent/rag → workflow → api → cmd`
3. 是否符合启动主链：`cmd → bootstrap → routes → handlers → domain`
4. `api/handlers` 是否只做协议适配
5. 是否收口到 `Builder/Factory/Registry`
6. 是否写清"保留/合并/下沉/删除"的具体对象
7. 是否涉及根目录治理

|------|------|----------|

| 命令 | 结果 |
|------|------|
| `go test ./pkg` | 通过 |

### 治理报告

- 完成项 + 证据

- gate 结果
- audit_plan verdict
- 明确："可归档" vs "可收尾"

## 4. 命令速查

| 场景 | 命令 |
|------|------|
| 格式检查 | `python scripts/refactor_plan_guard.py lint --target "<文件>" --require-tdd --require-verifiable-completion` |
| 进度报告 | `python scripts/refactor_plan_guard.py report --target "<文件>" --require-tdd --require-verifiable-completion` |
| 门禁 | `python scripts/refactor_plan_guard.py gate --target "<文件>" --require-tdd --require-verifiable-completion` |
| 归档审计 | `python .agents/skills/cn-refactor-plan-governance/scripts/audit_plan.py --target all` |
| 诊断扫描命令 | 见 `references/诊断命令参考.md` |
| 验证：构建 | `go build ./...` |

| 验证：单测 | `go test . -run "TestName" -count=1` |
| 验证：架构门禁 | `powershell -ExecutionPolicy Bypass -File scripts/arch_guard.ps1` |

## 5. 常见误判

### 诊断误判
- 只看目录不看代码 → 每个问题必须有文件+行号
- 定级随意 → P0=硬规则违反，P1=明确技术债
- 生成计划后不用户确认 → 必须等用户说可以

### 执行误判
- 只跑 lint 不改代码 → 必须改代码+验证
- 不排优先级直接改 → 先全局排序
- 改完不验证就勾 `[x]` → 验证不过不算完成
- 改完代码不更新计划 → 同步更新是强制要求
- 顺手做计划外改动 → 严格按计划范围
- 验证失败修无关问题 → 只修本轮引入的

### 治理误判
- 归档计划当基线
- 无证据标记 `[x]`
- 只写抽象建议不给具体路径
- 移动文件不删除旧实现
- 忽略 CLAUDE.md 架构约束
