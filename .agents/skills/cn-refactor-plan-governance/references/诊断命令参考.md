# 诊断命令参考

> 诊断模式扫描阶段的命令集合。按维度组织，可根据诊断范围选用。

---

## 1. 项目骨架

```bash
# 顶层目录结构
ls -la

# 所有包的目录树（3 层深度，排除 vendor/.git）
find . -maxdepth 3 -type d -not -path "*vendor*" -not -path "*.git*" -not -path "*node_modules*" | sort

# 按文件数统计包（识别过大包）
find . -maxdepth 3 -name "*.go" -not -path "*_test.go" -not -path "*vendor*" | \
  sed 's|/[^/]*$||' | sort | uniq -c | sort -rn | head -20
```

## 2. 大文件与 God Object

```bash
# 最大的 20 个源文件（排除测试和 vendor）
find . -name "*.go" -not -path "*_test.go" -not -path "*vendor*" -not -path "*.git*" | \
  xargs wc -l | sort -rn | head -20

# 超过 500 行的文件列表
find . -name "*.go" -not -path "*_test.go" -not -path "*vendor*" -not -path "*.git*" | \
  xargs wc -l | awk '$1 > 500 {print $2, $1" lines"}' | sort -rn

# import 超过 15 个包的文件（可能耦合过重）
for f in $(find . -maxdepth 4 -name "*.go" -not -path "*_test.go" -not -path "*vendor*"); do
  count=$(rg -c "^\s+\".*\"" "$f" 2>/dev/null || echo 0)
  [ "$count" -gt 15 ] && echo "$count imports: $f"
done
```

## 3. 依赖方向检查

对照 `CLAUDE.md` 中的分层规则：

```
Layer 0: types/   → 不依赖任何业务层
Layer 1: llm/     → 不依赖 agent/workflow/api/cmd
Layer 2: agent/ rag/ → 不依赖 cmd/
Layer 3: workflow/ → 不依赖 cmd/
适配层: api/      → 不承载业务决策
组合根: cmd/      → 不下沉业务实现
```

```bash
# agent/ 引用 cmd/（分层违反）
rg -l "\"github.com/BaSui01/agentflow/cmd" agent/ --type go

# workflow/ 引用 cmd/（分层违反）
rg -l "\"github.com/BaSui01/agentflow/cmd" workflow/ --type go

# api/handlers 有 gorm/infra 实现（适配层污染）
rg -l "gorm\.DB\|gorm\.Open\|sql\.Open" api/handlers/ --type go

# cmd/ 中直接 new handler/service（组合根越界）
rg -n "handlers\.New\|usecase\.New" cmd/ --type go

# api/handlers 直接依赖 domain 具体类型
rg -l "llm/core\|workflow/core\|workflow/observability" api/handlers/ --type go
```

## 4. 重复语义

```bash
# 搜索 Builder 模式（多入口迹象）
rg -n "type.*Builder interface" --type go
rg -n "func New.*Builder" --type go

# 搜索 Factory 模式
rg -n "type.*Factory interface" --type go
rg -n "func New.*Factory" --type go

# 搜索 Registry 模式
rg -n "type.*Registry interface" --type go
rg -n "func New.*Registry" --type go

# 搜索 Runtime 模式
rg -n "type.*Runtime interface" --type go
rg -n "type.*Runtime struct" --type go

# 搜索重复命名的核心概念
rg -n "type Team " --type go          # 多处定义 Team
rg -n "type Agent " --type go         # 多处定义 Agent
rg -n "type Config " --type go        # 多处定义 Config
rg -n "type Result " --type go        # 多处定义 Result
```

## 5. 公开表面与文档一致性

```bash
# README 中推荐的入口
rg -n "agent/runtime\|agent/team\|sdk/\|workflow/runtime" README.md README_EN.md docs --include "*.md" | \
  grep -v "docs/archive/" | grep -v "docs/重构计划/"

# 代码中实际可用的入口
rg -n "^func New\|^type.*Builder\|^func \(" --type go | head -40

# 文档中是否还在推荐旧路径
rg -n "agent/runtime\|agent/team\|teamadapter\|multiagent" README.md README_EN.md AGENTS.md docs/cn --include "*.md"
```

## 6. 架构守卫

```bash
# 找到所有守卫测试
rg -n "^func Test" architecture_guard_test.go

# 跑 architecture guard 测试
go test . -run "Test.*Guard\|Test.*Budget\|Test.*Allowlist\|Test.*Architecture" -count=1

# PowerShell 架构门禁
powershell -ExecutionPolicy Bypass -File scripts/arch_guard.ps1
```

## 7. 分层入口收敛

```bash
# 统计 sdk/ 公开入口
rg -n "^func New\|^type.*Builder" sdk/ --type go

# 统计 agent/runtime 公开入口
rg -n "^func New\|^func \(\w\+ \*" agent/runtime/ --type go

# 统计 agent/team 公开入口
rg -n "^func New\|^type.*interface" agent/team/ --type go

# 统计 workflow/runtime 公开入口
rg -n "^func New\|^type.*interface" workflow/runtime/ --type go
```
