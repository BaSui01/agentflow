## 🧭 建议的依赖顺序（目标）

1. `cmd/*`：只做启动、装配、配置注入，不承载业务。
2. `api/*`（HTTP adapter）：协议解析、鉴权入口、错误映射。
3. `app/*`（usecase）：业务编排（Agent/RAG/Workflow 场景）。
4. `domain/*`：实体、规则、接口。
5. `infra/*` 或 `pkg/*`：数据库、缓存、网络、第三方 provider 适配。
6. `types/*`：仅承载跨层稳定类型；禁止承载流程逻辑。

依赖约束：
- 仅允许“外层 → 内层”单向依赖。
- 禁止横向回跳（例如 `pkg -> api`、`domain -> cmd`）。
- 禁止兼容双轨：迁移时直接删除旧入口和旧路径。

## 📦 结构重组建议（针对当前仓库）

1. `agent` 重组
- 保留：`agent` facade（Builder/Config/public interfaces）。
- 下沉实现：`agent/runtime`、`agent/memory`、`agent/guardrails`、`agent/orchestration`。
- 结果：减少根包耦合与编译牵引面。

2. `llm/providers` 重组
- 保留每个 provider 子目录（这是合理 one/two-file adapter）。
- 将 `llm/factory` 改为注册式，避免中心 switch 持续增长。

3. `pkg` 重组
- 将 `pkg/middleware` 与 `api` 解耦，归位为纯基础设施。
- 评估合并 one-file 工具包到 `infra/platform`（仅对无独立边界者）。

4. `api` 重组
- 新增 `api/routes`，把路由表与 method/鉴权元数据集中。
- `api/handlers` 只保留 handler 与 DTO 转换，不承担路由拼装。
