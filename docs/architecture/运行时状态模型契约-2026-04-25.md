# 运行时状态模型契约（截至 2026-04-25）

本文档定义 AgentFlow 当前运行时状态模型的正式收口方向。它适用于单 Agent runtime、`agent/team` 与 `workflow/runtime`，用于 checkpoint、resume、replay、stream consumer、approval 与 audit 的统一语义。

## 正式契约归属

| 契约 | 正式类型 | 归属层 | 说明 |
|---|---|---|---|
| 运行配置 | `types.RunConfig` | Layer 0 `types/` | 模型、Provider、route policy、工具、预算、超时、metadata、tags 等 provider-neutral override。 |
| 运行状态 | `types.RunState` | Layer 0 `types/` | `run_id`、`session_id`、`checkpoint_id`、memory snapshot、tool state、pending approval 等可恢复状态 envelope。 |
| 运行事件 | `types.RunEvent` | Layer 0 `types/` | LLM chunk、tool call/result、handoff、approval、checkpoint、error、usage、workflow node 等统一事件语义。 |

## 当前落地状态

- `agent/runtime.RunConfig` 已收口为 `types.RunConfig` 的类型别名，继续保持原有 `agent/runtime` 使用方式。
- `types.RunConfig.ApplyToExecutionOptions(...)` 是运行时配置落到 `types.ExecutionOptions` 的唯一正式方法。
- `agent/observability/events.RuntimeStreamEvent.RunEvent()` 提供单 Agent stream event 到 `types.RunEvent` 的无损语义映射，不改变现有 stream JSON wire format。
- `workflow/core.WorkflowStreamEvent.RunEvent()` 提供 workflow stream event 到 `types.RunEvent` 的语义映射，不改变现有 workflow stream JSON wire format。
- `agent/core.Output.RunEvent(scope)` 可把 agent / team execution facade 输出映射为 `types.RunEvent`。
- `agent/runtime.TeamResult.RunEvent()` 可把 `agent/team` 官方 Team facade 结果映射为 `types.RunEvent`。

## 字段边界

- `RunConfig` 只承载执行期 override，不承载长期配置源；长期配置仍来自 `types.AgentConfig` / SDK options / server config。
- `RunState` 是可 checkpoint / resume 的状态 envelope，不替代具体 checkpoint store schema。
- `RunEvent` 是跨 runtime 的事件语义，不强制替换现有 runtime-specific stream DTO；对外兼容 DTO 可以保留，但必须能映射到 `RunEvent`。
- Provider SDK request / response struct 不允许进入 `RunConfig`、`RunState` 或 `RunEvent`。

## Resume / Replay / Audit 规则

### Resume

- 恢复入口必须先解析 `run_id`、`trace_id`、`checkpoint_id` 与 `session_id`；缺少 `checkpoint_id` 时才允许按 session / thread 查询 latest checkpoint。
- checkpoint 恢复只读取 `types.RunState` 可表达的状态：memory snapshot、tool state、pending approval、metadata 与 tags，不依赖隐式全局变量。
- 若存在 `pending_approval`，恢复后状态必须是 `RunStatusPendingApproval`，不得直接继续执行高风险工具。
- 恢复后的执行必须沿用原 `trace_id`，并可生成新的 child `run_id` / `parent_run_id` 串联恢复链。

### Replay

- replay 输入以 `RunEvent` 序列为准；runtime-specific stream DTO 必须先映射到 `types.RunEvent`。
- replay consumer 应按 `run_id` + `sequence` / `timestamp` 排序；缺少 sequence 时以 timestamp 为次优排序依据。
- tool result、approval、checkpoint、usage 事件必须保留原始 `tool_call_id` / `approval_id` / `checkpoint_id`，不得在 replay 中重写。
- replay 只能重放观测结果，不应再次执行工具、Provider 请求或审批动作。

### Audit

- audit 记录必须能从同一个 `run_id` 关联到 `trace_id`、principal、agent/team/workflow scope、tool、approval、checkpoint 与 usage。
- 授权和审批审计以 `AuthorizationService` 决策为准；tool runtime 只能补充执行结果，不重复做安全决策。
- 高风险 tool 的 audit event 必须带 `tool_call_id`、`tool_name`、risk、args fingerprint、decision 与可选 `approval_id`。
- audit sink 落账失败不得篡改运行结果；失败应形成错误日志或可观察事件，由上层按策略处理。

## 后续收口规则

- 新增 run metadata、tool state、approval state 字段时，优先加到 `types.RunState` 或 `types.RunEvent`，再由具体 runtime 做映射。
- 新增 stream event 类型时，必须补 `types.RunEventType` 映射和最小测试。
- resume / replay / audit 规则应以同一个 `run_id` 串起 `trace_id`、`checkpoint_id`、approval 与 usage。
