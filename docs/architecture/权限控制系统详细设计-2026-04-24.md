# 权限控制系统详细设计（package / 接口 / 数据结构级，2026-04-24）

> 本文是 `权限控制系统重构与引入方案-2026-04-24.md` 的实现级深化版本，目标是把方案继续细化到 **包布局、接口定义、数据结构、执行挂点、迁移切片**。

---

## 1. 设计目标

在不破坏当前主链的前提下，把现有：

- HTTP 鉴权
- Tool `PermissionManager`
- approval backend（复用历史 ToolApproval 能力）
- `HITL`
- grant/history store
- runtime approval events

收口成一套统一权限模型：

- **AuthN**：确认“谁”
- **AuthZ**：确认“能做什么”
- **Approval**：高风险动作的人类审批
- **Audit**：完整可追溯

---

## 2. 约束

### 2.1 必须遵守的架构约束

- Layer 0 `types/`：放零依赖契约
- `agent/`：运行时与能力层，不依赖 `cmd/`
- `workflow/`：编排层，可依赖 `agent/`
- `api/`：只做协议适配
- `cmd/`：只做组合装配

### 2.2 必须复用的现有能力

- `internal/app/bootstrap/http_auth_builder.go`
- `internal/app/bootstrap/authorization_policy_builder.go`
- `internal/app/bootstrap/authorization_approval_builder.go`
- `internal/app/bootstrap/agent_tooling_runtime_builder.go`
- `internal/usecase/tool_approval_service.go`
- `agent/integration/hosted/tools.go`
- `agent/observability/events/runtime_stream.go`

原则：

> **不新造一套并行权限系统；只在现有链路上抽统一模型与统一接口。**

#### `internal/usecase/authorization_service.go`

职责：

- 对外统一授权服务

## 5. 核心接口

## 5.1 PrincipalResolver

```go
type PrincipalResolver interface {
    ResolvePrincipal(ctx context.Context) (types.Principal, error)
}
```

用途：

- 从 HTTP auth context、A2A token、内部 service identity 中解析主体

## 5.2 PolicyEngine

```go
type PolicyEngine interface {
    Evaluate(ctx context.Context, req types.AuthorizationRequest) (*types.AuthorizationDecision, error)
}
```

用途：

- 核心策略判断
- 第一阶段可仅做 adapter 聚合，不必引入 DSL

## 5.3 ApprovalBackend

```go
type ApprovalBackend interface {
    RequestApproval(ctx context.Context, req types.AuthorizationRequest, preliminary *types.AuthorizationDecision) (*types.AuthorizationDecision, error)
    CheckApproval(ctx context.Context, approvalID string) (*types.AuthorizationDecision, error)
    Revoke(ctx context.Context, fingerprint string) error
}
```

用途：

- 封装现有 tool approval 能力，并以 `ApprovalBackend` 作为统一接口
- 未来扩到 shell / MCP / workflow step

## 5.4 AuditSink

```go
type AuditSink interface {
    RecordAuthorization(ctx context.Context, req types.AuthorizationRequest, decision *types.AuthorizationDecision) error
}
```

用途：

- 接 explainability / timeline / log / future DB audit

## 5.5 AuthorizationService

```go
type AuthorizationService interface {
    Authorize(ctx context.Context, req types.AuthorizationRequest) (*types.AuthorizationDecision, error)
}
```

推荐默认实现：

```go
type DefaultAuthorizationService struct {
    PrincipalResolver PrincipalResolver
    PolicyEngine      PolicyEngine
    ApprovalBackend   ApprovalBackend
    AuditSink         AuditSink
}
```

## 5.6 当前职责边界（截至 2026-04-25）

| 构件 | 归属层 | 当前职责 | 明确不负责 |
|---|---|---|---|
| `AuthorizationService` | `internal/usecase` | 唯一授权入口；解析 principal；调用 `PolicyEngine`；处理 `require_approval` 决策；调用 `AuditSink` 记录最终决策。 | 不直接执行工具；不直接读写 HITL pending map；不承载 HTTP 协议适配。 |
| `PolicyEngine` | `internal/usecase` interface + bootstrap adapter | 把 `AuthorizationRequest` 评估为 `AuthorizationDecision`；当前默认实现是 `toolPermissionPolicyEngine`。 | 不发起工具执行；不写审计；不直接暴露为用户入口。 |
| `PermissionManager` | `llm/capabilities/tools` | 作为 tool policy adapter 的下层规则引擎；匹配 agent/user/rule/role/condition；输出 allow/deny/require_approval。 | 不是新的官方授权入口；handler、chat、agent runtime 不应绕过 `AuthorizationService` 直接把它当上层准入服务。 |
| `ApprovalBackend` | `internal/usecase` interface + bootstrap adapter | `AuthorizationDecision=require_approval` 的统一后端抽象；当前默认 adapter 复用 `toolApprovalHandler`，可请求、查询、撤销 approval / grant。 | 不决定策略是否需要审批；不替代 `PolicyEngine`。 |
| `toolApprovalHandler` / approval manager | `internal/app/bootstrap/authorization_approval_builder.go` | 当前 hosted tool approval 的默认实现；负责 fingerprint、grant TTL、history，以及把 approval 请求落到 HITL interrupt；同时通过 `toolAuthorizationApprovalBackend` 接入 `AuthorizationService`。 | 不决定资源风险等级；不作为 handler 层直接执行入口。 |
| `HITL InterruptManager` | `agent/observability/hitl` | 管理 pending interrupt 生命周期：创建、resolve、reject、timeout、cancel、持久化状态。 | 不判断权限；不生成风险等级；不写 authorization audit。 |
| `AuditSink` | `internal/usecase` interface | 记录 `AuthorizationRequest` 与最终 `AuthorizationDecision`，当前默认 sink 写结构化日志，并把 `authorization_decision` 写入 `ToolApprovalHistoryStore`，包含 principal、agent、tool、fingerprint、run/trace、risk、approval id。 | 不改变决策；不发起审批；不执行工具。 |
| `AuthorizationAuditService` / `AuthorizationAuditHandler` | `internal/usecase` + `api/handlers` | 提供 `GET /api/v1/authorization/audit` 独立审计查询视图；复用现有 history store，只返回 `authorization_decision`，支持按 principal/user/agent/run/trace/resource/action/risk/decision/tool/fingerprint 过滤。 | 不写入审计；不解析策略；不复用 tool approval 的 HTTP 命名作为授权审计入口。 |
| hosted tool `ToolManager` | `internal/app/bootstrap/agent_tooling_runtime_builder.go` | 官方 chat/agent hosted tool 执行入口；执行前构造 `AuthorizationRequest` 并调用 `AuthorizationService`。 | 不承载策略规则；不绕开授权直接执行高风险工具。 |

当前 fail-closed 约束：

- `DefaultAuthorizationService` 无 `PolicyEngine` 时，`safe_read` 可 allow；`execution`、`networked_execution`、`mutating`、`admin`、`sensitive_read` 默认 deny。
- `toolPermissionPolicyEngine` 无 `PermissionManager` 时，safe-read 可 allow；高风险 tool resource 默认 deny。
- development/test 如需放行高风险工具，也必须显式配置 policy/rule；production 不允许因缺省策略静默放行高风险工具。

## 6. 和现有代码的映射关系

## 6.1 HTTP 鉴权

保留：

- `internal/app/bootstrap/http_auth_builder.go`

新增责任：

- 鉴权成功后，把 principal 写入 context

建议新增 helper：

- `types.WithPrincipal(ctx, principal)`
- `types.PrincipalFromContext(ctx)`

## 6.2 Tool 权限

保留：

- `PermissionManager`
- `PermissionRule`
- `PermissionContext`

新角色：

- 作为 `PolicyEngine` 的 **tool adapter**

即：

- `ResourceKind == tool | mcp_tool`
- `Action == execute`
- 转换为现有 `PermissionContext`
- 再转回 `AuthorizationDecision`

## 6.3 Tool Approval / HITL

保留：

- `internal/app/bootstrap/authorization_approval_builder.go`
- `toolApprovalRuntime`

新角色：

- `ApprovalBackend` 的默认实现

即：

- 当 `Decision == require_approval` 时
- 统一走 approval backend

## 6.4 Runtime 事件

扩展位置：

- `agent/observability/events/runtime_stream.go`
- `agent/runtime/request_runtime.go`

建议：

- 保留现有 approval 事件
- 再补统一 authorization metadata：
  - `decision`
  - `resource_kind`
  - `resource_id`
  - `policy_id`
  - `risk_tier`

## 7. 执行挂点设计

## 7.1 Tool Call 挂点（第一优先级）

挂点位置：

- `agent/integration/hosted/tools.go`
- `internal/app/bootstrap/agent_tooling_runtime_builder.go`

顺序：

1. 构造 `AuthorizationRequest`
2. 调 `AuthorizationService.Authorize(...)`
3. `allow` → 执行工具
4. `deny` → 返回拒绝
5. `require_approval` → 调 `ApprovalBackend`

## 7.2 Handoff 挂点

挂点位置：

- `internal/usecase/agent_service.go`
- 未来可落在 `agent/runtime` 的 handoff path

用途：

- 某 agent 能否 handoff 到另一个 agent
- 某 user 是否可触发高权限 subagent

## 7.3 Shell / Code Execution / File Write 挂点

挂点位置建议：

- shell：`agent/integration/hosted` 对应工具
- code execution：sandbox / hosted execution path
- file write：file ops tool

这些是第二批接入对象。

当前进展：

- [X] Workflow code step 已在 bootstrap 注入层通过 `AuthorizationService` 授权，资源为 `code_execution`，风险为 `RiskExecution`。
- [X] Workflow code step 授权上下文只记录 `language`、`code_bytes`、`code_fingerprint`、`timeout_seconds`、`max_code_bytes`、`max_timeout_seconds`、`max_output_bytes` 和 `allowed_languages`，不记录原始代码。
- [X] Workflow code step 执行前已校验 `input.code` 最大 64 KiB 与 `timeout_seconds` 最大 30 秒，sandbox 默认输出上限为 1 MiB。
- [X] `hosted_tools.shell` 启用后，`run_command` 只在 `AgentToolingRuntime` 中注册，执行路径为 `ToolManager -> AuthorizationService -> ToolRegistry.Execute -> ShellTool.Execute`。
- [X] `hosted_tools.file_ops` 启用后，`write_file` / `edit_file` 只在 `AgentToolingRuntime` 中注册，执行路径为 `ToolManager -> AuthorizationService -> ToolRegistry.Execute -> FileOpsTool.Execute`。
- [X] DB 动态 alias 指向 `run_command` / `write_file` / `edit_file` 时不降级成普通 `tool`，而是继承目标资源的 `ResourceKind` 与 `RiskTier`。
- [X] 架构守卫已限制服务代码直接构造 `hosted.NewShellTool(...)`、`hosted.NewWriteFileTool(...)`、`hosted.NewEditFileTool(...)`，避免高风险 hosted tools 绕过统一授权装配。

## 7.4 Workflow 节点挂点

挂点位置建议：

- `workflow/steps/orchestration.go`
- `workflow/runtime/*`

用途：

- 对高风险 workflow step 做授权或审批

## 8.1 默认 principal 角色

- `admin`
- `operator`
- `developer`
- `service`
- `agent_default`
- `agent_readonly`

## 8.2 默认资源策略

| 资源 | 默认策略 |
|---|---|
| `tool:web_search` / `tool:retrieval` / `file_read` | allow |
| `mcp_tool` | require_approval |
| `file_write` | require_approval |
| `shell_command` | require_approval |
| `code_execution` | require_approval |
| 未知工具 | deny |
| `admin_api` | deny（除非 admin/operator） |

## 8.3 环境策略

- `development`
  - approval 可放宽
  - allow_no_auth 仅在此环境允许
- `staging`
  - 接近生产
  - 高风险动作默认 require_approval
- `production`
  - fail-closed
  - 未知资源默认 deny

## 9. Grant / Scope 设计

保持兼容现有 3 种 scope：

- `request`
  - 适合最细粒度审批
  - `agent + resource + args hash`
- `agent_tool`
  - 同一 agent 同一资源可复用
- `tool`
  - 最宽范围，仅建议在可信环境使用

建议新增统一字段：

- `resource_kind`
- `resource_id`
- `principal_id`
- `agent_id`

这样以后同一套 grant store 能覆盖 tool / MCP / shell。
