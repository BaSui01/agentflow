# 权限控制系统详细设计（package / 接口 / 数据结构级，2026-04-24）

> 本文是 `权限控制系统重构与引入方案-2026-04-24.md` 的实现级深化版本，目标是把方案继续细化到 **包布局、接口定义、数据结构、执行挂点、迁移切片**。

---

## 1. 设计目标

在不破坏当前主链的前提下，把现有：

- HTTP 鉴权
- Tool `PermissionManager`
- `ToolApproval`
- `HITL`
- grant/history store
- runtime approval events

收口成一套统一权限模型：

- **AuthN**：确认“谁”
- **AuthZ**：确认“能做什么”
- **Approval**：高风险动作的人类审批
- **Audit**：完整可追溯

---

## 2. 约束

### 2.1 必须遵守的架构约束

- Layer 0 `types/`：放零依赖契约
- `agent/`：运行时与能力层，不依赖 `cmd/`
- `workflow/`：编排层，可依赖 `agent/`
- `api/`：只做协议适配
- `cmd/`：只做组合装配

### 2.2 必须复用的现有能力

- `internal/app/bootstrap/http_auth_builder.go`
- `internal/app/bootstrap/agent_tool_policy_builder.go`
- `internal/app/bootstrap/agent_tool_approval_builder.go`
- `internal/app/bootstrap/agent_tooling_runtime_builder.go`
- `internal/usecase/tool_approval_service.go`
- `agent/integration/hosted/tools.go`
- `agent/observability/events/runtime_stream.go`

原则：

> **不新造一套并行权限系统；只在现有链路上抽统一模型与统一接口。**

## 3. 推荐包布局

## 3.1 第一阶段（最小增量）

### Layer 0：公共契约

#### `types/authz.go`

职责：

- 权限系统零依赖契约
- 供 `api` / `agent` / `workflow` / `internal/usecase` 共享

放这里的类型：

- `PrincipalKind`
- `ResourceKind`
- `ActionKind`
- `DecisionKind`
- `RiskTier`
- `Principal`
- `AuthorizationRequest`
- `AuthorizationDecision`
- `ApprovalScope`
- `ApprovalRecord`

### Application / Usecase 层

#### `internal/usecase/authorization_service.go`

职责：

- 对外统一授权服务
- 聚合 principal resolver / policy engine / approval backend / audit sink
- 不关心 HTTP 或具体 tool registry 细节

#### `internal/usecase/authorization_context.go`

职责：

- 统一组装授权上下文
- 把 `trace_id` / `session_id` / `agent_id` / `workflow_id` / `user_id` 汇总成统一 map

### Bootstrap 层

#### `internal/app/bootstrap/authorization_builder.go`

职责：

- 创建默认 `AuthorizationService`
- 复用现有 `PermissionManager`
- 复用现有 `toolApprovalRuntime`
- 注入默认 audit sink

### Agent Runtime 层

#### `agent/execution/runtime/authorization_runtime.go`

职责：

- runtime 里的权限决策入口
- 提供 `AuthorizeToolCall(...)`
- 预留 `AuthorizeHandoff(...)` / `AuthorizeShellExecution(...)` / `AuthorizeWorkflowAction(...)`

### Workflow 层（后续）

#### `workflow/runtime/authorization_hooks.go`

职责：

- workflow 节点执行前的统一授权挂点
- 先不实现复杂逻辑，只定义接缝

---

## 3.2 第二阶段（资源类型扩展后）

当第二类审批资源（例如 shell / file write）稳定后，再考虑抽持久化层：

### `agent/persistence/authorization/`

可选子模块：

- `grant_store.go`
- `history_store.go`
- `policy_store.go`

当前阶段不建议先做，避免为了抽象而抽象。

## 5. 核心接口

## 5.1 PrincipalResolver

```go
type PrincipalResolver interface {
    ResolvePrincipal(ctx context.Context) (types.Principal, error)
}
```

用途：

- 从 HTTP auth context、A2A token、内部 service identity 中解析主体

## 5.2 PolicyEngine

```go
type PolicyEngine interface {
    Evaluate(ctx context.Context, req types.AuthorizationRequest) (*types.AuthorizationDecision, error)
}
```

用途：

- 核心策略判断
- 第一阶段可仅做 adapter 聚合，不必引入 DSL

## 5.3 ApprovalBackend

```go
type ApprovalBackend interface {
    RequestApproval(ctx context.Context, req types.AuthorizationRequest, preliminary *types.AuthorizationDecision) (*types.AuthorizationDecision, error)
    CheckApproval(ctx context.Context, approvalID string) (*types.AuthorizationDecision, error)
    Revoke(ctx context.Context, fingerprint string) error
}
```

用途：

- 封装现有 `ToolApproval`
- 未来扩到 shell / MCP / workflow step

## 5.4 AuditSink

```go
type AuditSink interface {
    RecordAuthorization(ctx context.Context, req types.AuthorizationRequest, decision *types.AuthorizationDecision) error
}
```

用途：

- 接 explainability / timeline / log / future DB audit

## 5.5 AuthorizationService

```go
type AuthorizationService interface {
    Authorize(ctx context.Context, req types.AuthorizationRequest) (*types.AuthorizationDecision, error)
}
```

推荐默认实现：

```go
type DefaultAuthorizationService struct {
    PrincipalResolver PrincipalResolver
    PolicyEngine      PolicyEngine
    ApprovalBackend   ApprovalBackend
    AuditSink         AuditSink
}
```

## 6. 和现有代码的映射关系

## 6.1 HTTP 鉴权

保留：

- `internal/app/bootstrap/http_auth_builder.go`

新增责任：

- 鉴权成功后，把 principal 写入 context

建议新增 helper：

- `types.WithPrincipal(ctx, principal)`
- `types.PrincipalFromContext(ctx)`

## 6.2 Tool 权限

保留：

- `PermissionManager`
- `PermissionRule`
- `PermissionContext`

新角色：

- 作为 `PolicyEngine` 的 **tool adapter**

即：

- `ResourceKind == tool | mcp_tool`
- `Action == execute`
- 转换为现有 `PermissionContext`
- 再转回 `AuthorizationDecision`

## 6.3 Tool Approval / HITL

保留：

- `internal/app/bootstrap/agent_tool_approval_builder.go`
- `toolApprovalRuntime`

新角色：

- `ApprovalBackend` 的默认实现

即：

- 当 `Decision == require_approval` 时
- 统一走 approval backend

## 6.4 Runtime 事件

扩展位置：

- `agent/observability/events/runtime_stream.go`
- `agent/execution/runtime/request_runtime.go`

建议：

- 保留现有 approval 事件
- 再补统一 authorization metadata：
  - `decision`
  - `resource_kind`
  - `resource_id`
  - `policy_id`
  - `risk_tier`

## 7. 执行挂点设计

## 7.1 Tool Call 挂点（第一优先级）

挂点位置：

- `agent/integration/hosted/tools.go`
- `internal/app/bootstrap/agent_tooling_runtime_builder.go`

顺序：

1. 构造 `AuthorizationRequest`
2. 调 `AuthorizationService.Authorize(...)`
3. `allow` → 执行工具
4. `deny` → 返回拒绝
5. `require_approval` → 调 `ApprovalBackend`

## 7.2 Handoff 挂点

挂点位置：

- `internal/usecase/agent_service.go`
- 未来可落在 `agent/execution/runtime` 的 handoff path

用途：

- 某 agent 能否 handoff 到另一个 agent
- 某 user 是否可触发高权限 subagent

## 7.3 Shell / Code Execution / File Write 挂点

挂点位置建议：

- shell：`agent/integration/hosted` 对应工具
- code execution：sandbox / hosted execution path
- file write：file ops tool

这些是第二批接入对象。

## 7.4 Workflow 节点挂点

挂点位置建议：

- `workflow/steps/orchestration.go`
- `workflow/runtime/*`

用途：

- 对高风险 workflow step 做授权或审批

## 8. 默认策略设计

## 8.1 默认 principal 角色

- `admin`
- `operator`
- `developer`
- `service`
- `agent_default`
- `agent_readonly`
- `agent_execution`

## 8.2 默认资源策略

| 资源 | 默认策略 |
|---|---|
| `tool:web_search` / `tool:retrieval` / `file_read` | allow |
| `mcp_tool` | require_approval |
| `file_write` | require_approval |
| `shell_command` | require_approval |
| `code_execution` | require_approval |
| 未知工具 | deny |
| `admin_api` | deny（除非 admin/operator） |

## 8.3 环境策略

- `development`
  - approval 可放宽
  - allow_no_auth 仅在此环境允许
- `staging`
  - 接近生产
  - 高风险动作默认 require_approval
- `production`
  - fail-closed
  - 未知资源默认 deny

## 9. Grant / Scope 设计

保持兼容现有 3 种 scope：

- `request`
  - 适合最细粒度审批
  - `agent + resource + args hash`
- `agent_tool`
  - 同一 agent 同一资源可复用
- `tool`
  - 最宽范围，仅建议在可信环境使用

建议新增统一字段：

- `resource_kind`
- `resource_id`
- `principal_id`
- `agent_id`

这样以后同一套 grant store 能覆盖 tool / MCP / shell。
