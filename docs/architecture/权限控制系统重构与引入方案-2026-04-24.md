# 权限控制系统重构与引入方案（截至 2026-04-24）

> 目标：在不推翻现有能力的前提下，把当前零散的 **HTTP 鉴权 / Tool 权限 / HITL 审批 / Grant TTL / 审计事件** 收口成一套统一、可扩展、可审计的权限控制系统。

---

## 1. 先说结论

这个项目**已经有权限控制的雏形**，不是从零开始：

- HTTP 鉴权入口已存在
- Tool `PermissionManager` 已存在
- 高风险工具审批 / 授权窗口 / 历史记录已存在
- Runtime stream / explainability 里已经有 approval 事件

所以正确路线不是“新起一套 authz 系统”，而是：

> **把当前 tool-level approval 能力纳入“统一授权系统”的第一个落点。**

核心思路：

1. 保留现有 `PermissionManager + approval backend + HITL` 能力，并统一挂到 `AuthorizationService` 主链
2. 补统一的 **Subject / Resource / Action / Context / Decision**
3. 把权限控制从“只管 hosted tool”扩展到：
   - tool
   - MCP
   - shell / code execution
   - file write
   - workflow execution
   - handoff / subagent
   - 管理类 API
4. 统一审计、审批、授权窗口、撤销、过期、事件语义

## 2.2 当前问题

### 问题 1：权限系统曾经偏向 tool-level approval

虽然功能已经不少，但当前公共语义仍偏向：

- “某个 hosted tool 要不要审批”

还没有抽象成完整授权模型：

- 谁（subject）
- 对什么（resource）
- 做什么（action）
- 在什么上下文（context）
- 得到什么决策（decision）

### 问题 2：鉴权和授权还没有统一上下文

现状上 HTTP 鉴权、runtime 工具执行、handoff、workflow 节点执行，各自带的身份上下文还不够统一。

结果：

- `user_id`、`agent_id`、`trace_id`、`session_id` 已零散存在
- 但没有统一的 `AuthorizationContext`

### 问题 3：资源类型还不完整

当前最成熟的是 tool。

但真正要做权限系统，至少还要覆盖：

- `tool`
- `mcp_tool`
- `shell_command`
- `code_execution`
- `file_read`
- `file_write`
- `workflow_execute`
- `handoff`
- `admin_api`

### 问题 4：决策结果模型不够统一

当前实际存在的语义包括：

- allow
- deny
- requires_approval

但没有统一到整个系统的公共类型里。

## 3. 目标架构

## 3.1 统一权限控制模型

建议引入统一术语：

### Subject（主体）

谁在发起操作：

- user
- api key
- service
- agent
- subagent
- workflow node

### Resource（资源）

操作目标：

- tool
- file
- shell
- mcp server / mcp tool
- workflow
- conversation / session
- admin API

### Action（动作）

操作类型：

- read
- execute
- write
- delete
- route
- handoff
- approve
- manage

### Context（上下文）

决策上下文：

- user_id
- agent_id
- team_id
- workflow_id
- trace_id
- session_id
- environment
- risk tier
- request metadata

### Decision（决策）

统一结果：

- `allow`
- `deny`
- `require_approval`

---

## 3.2 目标分层

### 层 1：Authentication（确认你是谁）

入口：

- HTTP middleware
- A2A / MCP / internal service call

输出：

- `AuthPrincipal`

### 层 2：Authorization（确认你能做什么）

核心服务：

- `AuthorizationService`
- `PolicyEngine`

输入：

- principal
- resource
- action
- context

输出：

- `AuthorizationDecision`

### 层 3：Approval（高风险动作的人类审批）

现有链路直接复用：

- approval backend（复用历史 ToolApproval 能力）
- `HITL`
- grant store

统一定位：

- 它不是单独系统
- 它是 `AuthorizationDecision = require_approval` 的执行后端

### 层 4：Audit / Observability（审计与追踪）

统一记录：

- who
- what
- decision
- why
- approval id / grant id
- trace id

## 4. 推荐数据结构

## 4.1 Principal

```go
type Principal struct {
    Type      string            // user | api_key | service | agent
    ID        string
    Roles     []string
    TenantID  string
    Metadata  map[string]string
}
```

## 4.2 AuthorizationRequest

```go
type AuthorizationRequest struct {
    Principal Principal
    ResourceType string      // tool, file, shell, workflow, api
    ResourceID   string
    Action       string      // read, execute, write, approve, manage
    Context      map[string]any
}
```

## 4.3 AuthorizationDecision

```go
type AuthorizationDecision struct {
    Decision   string // allow | deny | require_approval
    Reason     string
    PolicyID   string
    ApprovalID string
    ExpiresAt  *time.Time
}
```

## 5. 如何挂到当前代码链路

## 5.1 第一阶段：不要动 HTTP 鉴权，先抽授权服务

做法：

- 保留 `BuildAuthMiddleware(...)`
- 把鉴权结果注入一个统一 principal context
- 新增 `AuthorizationService`

建议新增文件：

- `internal/usecase/authorization_service.go`
- `internal/app/bootstrap/authorization_builder.go`
- `types/authz.go` 或 `agent/core/authz.go`

## 5.2 第二阶段：把 Tool 权限提升成通用授权后端

现有：

- `PermissionManager`
- `PermissionContext`
- `PermissionRule`

建议：

- 不删除
- 但让它成为 `AuthorizationService` 的一个 adapter

即：

- tool 类资源 → 继续走现有 `PermissionManager`
- 其他资源 → 未来补其他 adapter

## 5.3 第三阶段：把 tool approval 能力正式定位为 Approval Backend

现有：

- `toolApprovalHandler`
- `toolApprovalRuntime`
- grant store / history store

建议：

- 保留现有实现
- 但语义命名上，把它升级为通用 approval backend 的第一种实现

未来可以扩展：

- tool approval
- MCP approval
- shell approval
- workflow step approval

## 5.4 第四阶段：把风险分级做成标准字段

当前已存在：

- `hosted_tool_risk`

建议统一风险等级：

- `safe_read`
- `sensitive_read`
- `mutating`
- `execution`
- `networked_execution`
- `admin`

所有高风险资源都必须带 risk tier。

## 5.5 第五阶段：补角色与策略模型

建议最小角色：

- `admin`
- `operator`
- `developer`
- `service`
- `agent_default`
- `read_only_agent`
- `execution_agent`

策略可先支持：

- allow list
- deny list
- wildcard match
- environment gate
- approval-required gate

## 6. 推荐迁移切片

## Slice-1：统一术语与公共类型

- [ ] 新增 `Principal` / `AuthorizationRequest` / `AuthorizationDecision`
- [ ] 鉴权 middleware 输出 principal context
- [ ] tool permission adapter 接到统一授权接口

验收：

- 不改现有 API 行为
- 只是补一层统一模型

## Slice-2：把 tool approval 语义提升为 authorization approval

- [ ] `require_approval` 明确成为标准决策结果
- [ ] stream event / explainability / history 统一使用相同 decision 语义
- [ ] 文档统一写法：approval 是 authorization backend，不是独立临时机制

## Slice-3：扩资源类型

- [ ] 支持 `shell_command`
- [ ] 支持 `code_execution`
- [ ] 支持 `file_write`
- [ ] 支持 `mcp_tool`
- [ ] 支持 `handoff`

## Slice-4：角色与环境隔离

- [ ] 不同环境策略（dev / staging / prod）
- [ ] 不同 principal 角色默认策略
- [ ] admin API 与 runtime action 分层

## Slice-5：统一审计与回放

- [ ] 统一 authorization audit entry
- [ ] trace / approval / grant / revoke 可串联
- [ ] 支持查询某次执行的完整权限决策链

## 7. 验收标准

### 功能

- [ ] 同一套授权模型能覆盖 HTTP、tool、MCP、shell、workflow、handoff
- [ ] `require_approval` 成为标准能力，不再只是 hosted tool 特例
- [ ] grant/revoke/ttl/history/audit 语义统一

### 架构

- [ ] 不新增并行权限系统
- [ ] 现有 `PermissionManager` / approval backend / `HITL` 被复用，而不是绕开
- [ ] 鉴权与授权上下文统一

### 可观测性

- [ ] runtime stream 可见 decision / approval / grant
- [ ] explainability timeline 可回放完整授权链
- [ ] 审计日志能按 user / agent / tool / trace 维度追查
