# 我的 Agent 框架设计参考（扩展版，含更多框架调研）

> 目标：基于截至 **2026-04-23** 的主流 Agent 框架官方资料，提炼一份适合 **AgentFlow/Go 项目** 的自定义 Agent 框架设计参考。

---

## 1. 设计结论先行

如果要做你自己的 Agent 框架，我建议不是“模仿某一家”，而是做一个**组合式架构**：

- **OpenAI Agents SDK** 学最小 runtime 原语
- **LangGraph / Microsoft Agent Framework** 学显式编排、状态图、HITL、checkpoint
- **Google ADK** 学 workflow agents、Session/State/Memory、Go 友好 API
- **Semantic Kernel** 学 enterprise middleware / plugins / orchestration patterns
- **PydanticAI** 学类型安全、依赖注入、输出校验、durable execution
- **CrewAI** 学 Flow 与 Crew 的清晰职责分离
- **LlamaIndex** 学 event-driven workflow 与 data-centric agent
- **Mastra / Agno** 学 agents + workflows + runtime + production plane 一体化形态
- **smolagents / Pi** 学 coding-agent harness、code agent、RPC/headless、local-first session

一句话设计目标：

> **做一个“单 Agent runtime 足够轻、多 Agent facade 足够稳、workflow 编排足够强、工具/MCP/A2A 足够开放、状态与事件足够统一”的框架。**

## 2. 扩展框架调研（官方资料）

## 2.1 OpenAI Agents SDK

核心印象：

- 原语少
- handoff 是一等能力
- tracing / sessions / MCP / human-in-the-loop 都围绕 runtime 展开

适合借鉴：

- `Agent / Tool / Handoff / Session / Trace` 作为最小原语
- streaming event 语义标准化
- 把“agent loop”做成统一、可组合的核心

不建议照搬：

- 不要把 workflow 编排塞回单 Agent runtime

参考：

- https://openai.github.io/openai-agents-python/
- https://openai.github.io/openai-agents-python/multi_agent/

## 2.2 Google ADK

官方当前给出的核心形态非常适合做 Go 参考：

- 支持 **Python / TypeScript / Go / Java**
- 区分：
  - `LlmAgent`
  - workflow agents：`SequentialAgent` / `ParallelAgent` / `LoopAgent`
  - custom agent
- 明确提供：
  - Session / State / Memory
  - Artifacts
  - graph-based workflows
  - Agent Registry
  - A2A / MCP 生态接入

适合借鉴：

- **workflow agents** 的建模方式
- **Session / State / Memory** 的显式拆分
- 多语言一致的 API 表面

参考：

- https://adk.dev/
- https://adk.dev/get-started/about/
- https://adk.dev/agents/
- https://adk.dev/agents/workflow-agents/
- https://adk.dev/workflows/
- https://adk.dev/sessions/

## 2.3 Microsoft Agent Framework

截至 **2026-02-20**，Microsoft Learn 明确写明：

- 它是 **AutoGen + Semantic Kernel** 的直接后继
- 组合了：
  - AutoGen 的 agent abstraction
  - Semantic Kernel 的 session state / type safety / middleware / telemetry
  - graph-based workflows

适合借鉴：

- **Agent vs Workflow** 的硬边界
- enterprise runtime（middleware / telemetry / state）
- graph-based orchestration

参考：

- https://learn.microsoft.com/en-us/agent-framework/overview/
- https://learn.microsoft.com/en-us/agent-framework/
- https://learn.microsoft.com/en-us/agent-framework/migration-guide/

## 2.4 Semantic Kernel

Semantic Kernel 仍然很值得参考，因为它在企业集成层面成熟：

- plugin / function calling / middleware
- 多语言 SDK
- Agent Orchestration 中提供：
  - Concurrent
  - Sequential
  - Handoff
  - Group Chat
  - Magentic
- runtime 被明确定义为 message bus + lifecycle + execution context

适合借鉴：

- plugin / tool / service container 模式
- orchestration pattern SDK 化
- runtime 与 actor lifecycle 的清晰定义

注意：

- Microsoft 现在主推的是 Agent Framework
- Semantic Kernel 的 orchestration 文档仍标记有 experimental 阶段内容

参考：

- https://learn.microsoft.com/en-us/semantic-kernel/overview/
- https://learn.microsoft.com/en-us/semantic-kernel/frameworks/agent/agent-orchestration/

## 2.5 AutoGen（历史重要，但不建议作为未来主参考）

官方当前已明确：

- **AutoGen is now in maintenance mode**
- 建议迁移到 Microsoft Agent Framework

但它仍值得参考，因为很多多 Agent 设计范式就是它带火的：

- GroupChat
- event-driven runtime
- low-level core + high-level team dual model

适合借鉴：

- 历史模式与经验教训

不建议照搬：

- 不再作为未来框架基线

参考：

- https://github.com/microsoft/autogen
- https://learn.microsoft.com/en-us/agent-framework/migration-guide/from-autogen/

## 2.6 LangChain / LangGraph / Deep Agents

LangChain 当前的 Agent 体系已经明确建立在 LangGraph runtime 之上：

- agent runtime 是 graph-based
- 多 Agent 模式明确区分：
  - subagents
  - handoffs
  - skills
  - router
  - custom workflow
- Deep Agents 进一步强调：
  - planning
  - subagents
  - file systems
  - long-term memory
  - harness

适合借鉴：

- **多 Agent 不是默认答案**，而是多种模式选择
- context engineering 是多 Agent 设计中心
- “agent harness” 这个概念很适合 coding agent

参考：

- https://docs.langchain.com/oss/python/langchain/agents

## 2.7 LlamaIndex

LlamaIndex 当前对“workflow + agent”这条线已经很清楚：

- workflow 是 **event-driven** abstraction
- step 处理事件并发出新事件
- 可以用 workflow 构建 agent、RAG flow、extraction flow
- 预置 `FunctionAgent` 和 `AgentWorkflow`

适合借鉴：

- event-driven workflow
- 把 agent 看成 workflow 中的节点，而不是唯一主角
- data/RAG 与 agent 的天然耦合表达

参考：

- https://docs.llamaindex.ai/en/stable/module_guides/workflow/
- https://docs.llamaindex.ai/en/stable/understanding/workflows/
- https://docs.llamaindex.ai/en/stable/understanding/agent/
- https://docs.llamaindex.ai/en/stable/understanding/agent/multi_agent/

## 2.8 CrewAI

CrewAI 的价值不在底层 runtime，而在产品建模方式：

- **Flow**：structured, event-driven workflow，负责 state 与 control flow
- **Crew**：autonomous team，负责 agent 协作
- **Process**：Sequential / Hierarchical / hybrid 等任务执行策略

适合借鉴：

- Flow 管流程、Crew 管自治
- 状态管理不能和 agent team 语义揉在一起

参考：

- https://docs.crewai.com/introduction
- https://docs.crewai.com/en/concepts/flows
- https://docs.crewai.com/en/concepts/processes
- https://docs.crewai.com/en/concepts/agents

## 2.9 PydanticAI

PydanticAI 当前最有价值的点在于：

- typed agent output
- model-agnostic
- MCP 多接法
- durable execution（官方直接支持 Temporal / DBOS / Prefect）
- multi-agent patterns 文档非常清晰：
  - delegation
  - hand-off
  - graph based control flow
  - deep agents

适合借鉴：

- 输出类型校验
- deps 注入
- durable execution 走公共接口
- usage / limits / observability 与 agent run 紧耦合

参考：

- https://ai.pydantic.dev/agent/
- https://ai.pydantic.dev/multi-agent-applications/
- https://ai.pydantic.dev/durable_execution/overview/
- https://ai.pydantic.dev/mcp/

## 2.10 Agno

Agno 的文档把系统拆成三层，很有产品架构感：

- Framework：agents / teams / workflows / memory / knowledge / guardrails / integrations
- Runtime：生产服务化 backend
- Control Plane：测试、监控、管理

适合借鉴：

- **框架层 / runtime 层 / control plane 层** 三层分离
- 把“怎么写 agent”和“怎么管生产系统”拆开

参考：

- https://docs.agno.com/
- https://docs.agno.com/introduction

## 2.11 Mastra

Mastra 当前非常像“现代 TS 全家桶 agent framework”：

- stateful agents
- workflows with graph / branch / loops / suspend / resume
- memory
- MCP client / server
- snapshots
- tracing / eval

适合借鉴：

- 把 workflow state persistence 做成一等公民
- agents / tools / workflows / MCP 的统一开发体验

参考：

- https://mastra.ai/agents
- https://mastra.ai/workflows
- https://mastra.ai/docs/agents/mcp-guide
- https://mastra.ai/en/reference/workflows/snapshots

## 2.12 Hugging Face smolagents

smolagents 的特点非常鲜明：

- 追求极简抽象
- 区分 `CodeAgent` 与 `ToolCallingAgent`
- 支持 sandboxed code execution
- model/tool/modality agnostic

适合借鉴：

- coding agent 与普通 tool-calling agent 分轨
- 用代码作为 agent action 的一等形态

参考：

- https://huggingface.co/docs/smolagents/index

## 2.13 Pi（pi-mono / pi coding agent）

`pi-mono` 官方仓库把自己描述为：

- AI agent toolkit
- coding agent CLI
- unified LLM API
- TUI / web UI libraries
- Slack bot

而 `packages/coding-agent/docs/rpc.md` 明确提供：

- **RPC headless mode**
- 可以把 coding agent 嵌入 IDE / UI / 其他应用
- 也可以直接使用 `AgentSession`

适合借鉴：

- coding-agent harness 形态
- local-first session 与 agent directory
- headless / RPC / embedded SDK 三种运行方式并存

注意：

- Pi 更像 **coding agent toolkit / harness**
- 不是通用企业 Agent Framework 的直接替代

参考：

- https://github.com/badlogic/pi-mono
- https://github.com/badlogic/pi-mono/blob/main/packages/coding-agent/docs/rpc.md

## 3. 归纳：我建议你的框架借什么、不借什么

| 框架 | 借什么 | 不借什么 |
|---|---|---|
| OpenAI Agents SDK | 最小原语、handoff、events、sessions、tracing、MCP、guardrails | 把一切编排都塞回 runtime |
| Google ADK | workflow agents、Session/State/Memory、多语言一致性、Go API 组织 | Google 生态特有的基础设施假设 |
| Microsoft Agent Framework | agent/workflow 分离、typed workflow、middleware、telemetry、enterprise runtime | 过重的生态绑定 |
| Semantic Kernel | plugins、middleware、runtime actor 心智 | 过多企业概念侵入核心 API |
| LangChain/LangGraph | 多 Agent 模式矩阵、middleware、durable runtime、custom workflow | 过度抽象导致的学习成本 |
| LlamaIndex | RAG-first agent、workflow event、数据上下文组织 | 让数据索引层吞掉 agent/runtime 边界 |
| Haystack | pipeline/component + agentic RAG | 把 pipeline component 体系照搬成顶层架构 |
| CrewAI | Flow vs Crew 分工、角色任务表达 | 让团队模型反向吞掉流程层 |
| PydanticAI | typed output、deps、durable execution 接口、tool/result validation | Python 特有的泛型/校验风格直接照搬 |
| Agno | agents/teams/workflows/memory/knowledge 的产品表面组织 | 大而全的公开 API 面 |
| Mastra | snapshots、MCP 一等支持、stateful workflows、observability/eval 产品面 | JS 生态实现细节直接平移 |
| Dapr Agents | actor/workflow/state store 的 durable 基建抽象 | 引入 Dapr runtime 作为硬依赖 |
| VoltAgent | DX、server/control plane、observability UI、workflow 可视化 | TS-first 框架结构 |
| BeeAI | provider-agnostic、memory/workflow/tool 的统一组合 | 未成熟生态的具体 API |
| smolagents | CodeAgent、code-as-action、managed agents 的简洁性 | 把通用 agent 全部变成代码执行器 |
| CAMEL / MetaGPT | role protocol、SOP、work product pipeline | research/app framework 的固定角色剧本 |
| Pi | coding-agent harness、RPC、embedded session | 把通用框架彻底做成 coding-agent 专用 |

### 3.1 融合路线：把外部能力映射到本项目唯一入口

外部框架能力进入本项目时，必须先映射到既有官方入口，而不是新建平行入口：

| 外部能力名 | 本项目落点 | 说明 |
|---|---|---|
| Agent / Assistant / LlmAgent | `agent/runtime` | 单 Agent 的模型循环、工具调用、handoff、guardrail、stream event |
| Crew / Team / Society / GroupChat | `agent/team` | 自治协作、多角色、多 Agent 任务分派；不承载确定性流程控制 |
| Flow / Graph / Workflow / Process / Pipeline | `workflow/runtime` | 顺序、并行、分支、循环、HITL、checkpoint、resume |
| Session / Conversation / Thread | `agent/runtime.Session` 或 SDK runtime session store | 长期对话上下文，不等于 checkpoint |
| State / Snapshot / Checkpoint | `workflow/runtime` + runtime checkpoint contract | 可恢复执行状态，必须可序列化、可审计 |
| Memory / Knowledge / Index | `agent/capabilities/memory` + `rag/` | 记忆和知识检索作为上下文/工具能力，不反向依赖 API/cmd |
| Tool / Function / Action / Component | 统一 tool contract | tool args/result typed schema、permission、audit、retry 统一处理 |
| Middleware / Guardrail / Policy | runtime hooks + `internal/usecase/authorization_service.go` | 模型前后、工具前后、输出前后的统一拦截点 |
| Trace / Eval / Observability | `agent/observability` | 消费 `RunEvent`，不控制核心执行流 |
| Coding Agent Harness | 独立 harness | shell/file/diff/browser/subagent 是专线能力，不能污染通用 Agent API |

## 4. 我的框架目标形态

## 4.1 四层模型

### 第 1 层：Surface（入口层）

只保留四个官方表面：

- `sdk`：统一装配入口
- `agent/runtime`：高级单 Agent runtime
- `agent/team`：官方多 Agent facade
- `workflow/runtime`：显式编排层

### 第 2 层：Execution Core（执行内核）

最小核心原语：

- `Agent`
- `Tool`
- `Handoff`
- `Guardrail`
- `Session`
- `RunConfig`
- `RunEvent`

### 第 3 层：State & Protocol（状态与协议）

- Session / State / Memory
- Checkpoint / Resume
- MCP / A2A
- Artifact / File / Sandbox

### 第 4 层：Ops & Evaluation（运维与评估）

- tracing
- metrics
- eval
- audit log
- policy / approval

---

## 4.2 单 Agent 与 Workflow 的边界

### 单 Agent 负责

- 理解任务
- 调用工具
- 自治 loop
- handoff
- 结构化输出
- 发出标准事件

### Workflow 负责

- 顺序、并行、分支、循环、子图
- 跨 agent / 跨 tool / 跨人工审批的控制流
- checkpoint / resume
- deterministic orchestration

这条边界是整个设计里最重要的部分。

---

## 4.3 Team 与 Multi-Agent 的边界

### Team（公开表面）

职责：

- 给用户一个稳定 facade
- 暴露有限、可理解的协作模式
- 保证输出/事件/错误语义统一

### Multi-Agent Engine（内部实现）

职责：

- debate / router / broadcast / handoff / parallel / loop 等具体模式实现
- 作为 `team` 和 `workflow` 的内部执行器

原则：

- **用户看到 team**
- **内部自由演化 multi-agent engines**

## 5.2 推荐核心接口（Go 草图）

```go
type Agent interface {
    ID() string
    Run(ctx context.Context, input *RunInput) (*RunOutput, error)
    Stream(ctx context.Context, input *RunInput, emit EventEmitter) error
}

type Tool interface {
    Name() string
    Schema() ToolSchema
    Execute(ctx context.Context, call ToolCall) (ToolResult, error)
}

type Team interface {
    ID() string
    Execute(ctx context.Context, task string, opts ...TeamOption) (*TeamResult, error)
}

type Workflow interface {
    ID() string
    Run(ctx context.Context, input any) (any, error)
    Resume(ctx context.Context, checkpointID string, input any) (any, error)
}
```

## 6. 我会怎么设计“统一运行时状态”

## 6.1 RunConfig

每次调用可覆盖：

- model
- provider
- route policy
- budget
- max iterations
- allowed tools
- tags / metadata

## 6.2 Session

长期上下文与连续对话：

- session id
- messages
- user-scoped memory links
- active agent/team/workflow identity

## 6.3 Checkpoint

可恢复状态：

- current node / current stage
- pending tool calls
- pending approvals
- workflow graph cursor
- resumable snapshot

## 6.4 RunEvent

统一事件模型建议至少覆盖：

- token
- reasoning
- tool_call
- tool_result
- handoff_requested
- handoff_completed
- approval_requested
- approval_resolved
- checkpoint_saved
- state_changed
- completed
- failed

## 7. 我会怎么设计“coding agent 专线”

如果你以后想做自己的 coding agent，我建议不要强行把它塞进普通聊天 agent。

应该单独做一个 **CodingAgent Harness**：

- 文件系统能力
- diff / patch / apply
- shell / sandbox
- subagent delegation
- plan / task list / progress
- headless RPC mode
- terminal UI
- browser / MCP tool bridge

这部分最值得参考：

- OpenAI Agents SDK（sandbox / streaming / handoff）
- LangChain Deep Agents（agent harness）
- smolagents（CodeAgent）
- Pi（coding agent CLI + AgentSession + RPC）

## 8. 我建议你自己的框架优先做什么

### 第一阶段：先做“稳”

- 单 Agent runtime
- tool calling
- structured output
- MCP
- stream events
- session

### 第二阶段：再做“多 Agent”

- team facade
- supervisor / parallel / handoff
- subagent isolation

### 第三阶段：再做“强编排”

- workflow graph
- branch / loop / checkpoints
- human approval
- durable execution

### 第四阶段：最后做“平台化”

- eval
- observability
- control plane
- registry / marketplace

---

## 9. 最终设计原则（建议直接采纳）

1. **单一官方入口**
2. **单 Agent runtime 与 workflow 编排分层**
3. **team 是唯一官方多 Agent 表面**
4. **tool / MCP / A2A 使用统一 contract**
5. **状态、事件、checkpoint 必须标准化**
6. **coding agent 是专线，不是普通聊天 agent 的补丁**
7. **先做可验证内核，再做 control plane**
