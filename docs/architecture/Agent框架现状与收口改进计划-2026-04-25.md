# Agent 框架现状与收口改进计划（截至 2026-04-25）

## 1. 结论

AgentFlow 当前不是空壳，也不是只能演示的原型。项目已经具备可运行的 Agent 框架内核：统一 SDK 装配、单 Agent runtime、function calling / tool calling、多 Agent team、workflow runtime、RAG、Hosted Tools、MCP 工具桥接、工具审批和基础授权抽象都已经落到代码里。

但当前也还不是“收口完成态”。主要问题不是缺少某一个大功能，而是公开入口、内部执行入口、运行时状态、权限链路、文档示例还没有完全统一。当前更准确的判断是：

- 功能层：可用。
- 框架层：已经成形。
- 对外产品化层：仍需收口。
- 后续重点：减少并行入口，统一状态与授权链路，压缩 public surface，补齐回归矩阵。

综合判断：当前 Agent 框架约为 7/10 的可用成熟度，适合继续切片式收口，不建议推倒重写。

## 4. 当前主要问题

### 4.1 官方入口与内部执行入口已经初步收口

当前文档和规则已经明确正式入口，服务端 usecase、workflow orchestration、runtime orchestration 与 capability catalog 已收口到 `agent/team.ExecuteAgents(...)`、`team.GlobalModeExecutor()` 或 `ModeExecutor` 注入。内部 `multiagent` registry 仍存在，但只作为 `agent/team` 包内实现细节使用。

风险：

- 后续新增多 Agent 能力时仍可能把内部 engine API 重新 re-export。
- examples / livecheck 如果缺少守卫，容易再次示范内部构造器。
- `ModeRegistryExecutor` 等历史命名仍可能造成理解噪音，后续可按 P9/ADR 继续命名清理。

### 4.2 `agent/team` public surface 已缩小，仍需长期守卫

`agent/team/team.go` 已移除对 multiagent / hierarchical 内部类型和构造函数的 re-export；SharedState 通过 `agent/team` 自己的小接口与包装实现暴露，execution modes 通过 `ExecutionMode` 与 `ExecuteAgents(...)` 暴露。

风险：

- 后续新增能力时可能重新把内部 engine 类型作为便捷出口暴露。
- 历史示例或 livecheck 若绕过 guard，仍可能重新引入旧构造器。
- team / workflow / runtime orchestration 的命名仍带有部分历史痕迹，需要通过 ADR 和架构文档继续固化。

### 4.3 Runtime 大文件热点明显

当前热点文件行数约如下：

| 文件 | 行数 |
|---|---:|
| `agent/runtime/agent_builder.go` | 2982 |
| `agent/runtime/interfaces_runtime.go` | 2255 |
| `agent/runtime/registry_runtime.go` | 1591 |
| `agent/team/internal/engines/multiagent/multi_agent.go` | 1495 |
| `agent/runtime/request_runtime.go` | 651 |

风险：

- 单次修改影响面大。
- review 和测试定位成本高。
- 新能力容易继续塞进已有大文件。

### 4.4 运行时状态模型未统一

当前 RunConfig、session、checkpoint、memory、trace、usage、approval、stream event 分布在多个层次。

风险：

- resumable execution 难以形成统一契约。
- team / workflow / single agent 的状态传递容易不一致。
- 审批中断、恢复执行、trace 串联、成本落账之间缺少统一上下文对象。

### 4.5 授权链路仍需硬化

`AuthorizationService` 已经是正确方向，但当前默认 allow 适合开发期，不适合作为生产安全基线。

风险：

- 高风险工具如果绕过统一授权入口，审批和审计会出现空洞。
- 文件写入、shell、code execution、MCP 等工具需要更明确的风险等级与默认策略。
- PermissionManager、ApprovalBackend、HITL、AuditSink 容易形成多套判断点。

### 4.6 文档仍有历史口径混杂

当前 `docs/archive/` 已明确为历史追溯材料，但中文导航中仍把部分归档文档放在较靠前的位置。归档文档可以保留，但正式开发和评审时必须以 `README`、`AGENTS.md`、`docs/architecture/` 的当前专题文档为准。

风险：

- 新人容易把归档计划当成当前契约。
- “legacy 多 Agent”与“official team facade”的边界被弱化。

- [x] 将工具执行准入统一挂到 `AuthorizationService`。

- [x] 确保 chat tool path、agent tool path、MCP hosted tool path、retrieval tool path 都有一致授权和审计。
- [x] 增加审批 grant TTL、拒绝、超时、重复调用、并发审批的回归测试。

已完成的 P5 子项：

- `toolApprovalHandler.RequestApproval(...)` 对同一 approval fingerprint 串行执行 lookup/create/remember，避免并发重复创建多个 pending HITL interrupt。
- `AgentToolingRuntime` 持有统一 `AuthorizationService`，默认由同一个 `PermissionManager` 构建；server 启动与 hot reload 的 workflow runtime 复用该服务，避免再临时构造第二套授权入口。
- `hostedToolManager.ExecuteForAgent(...)` 在调用 hosted registry 前先执行 `AuthorizationService.Authorize(...)`；chat tool path 与 agent tool path 因共享同一个 runtime `ToolManager`，准入与审计入口保持一致。
- MCP hosted tool 与 retrieval tool 的授权请求会分别映射到 `ResourceMCPTool` / `ResourceTool`，并携带 `agent_id`、`tool_call_id`、`args_fingerprint`、`trace_id`、`run_id`、`hosted_tool_type`、`hosted_tool_risk` 等审计上下文。
- `DefaultAuthorizationService` 在无 `PolicyEngine` 时采用更严格的默认语义：`safe_read` 可 allow，高风险 `execution` / `networked_execution` / `mutating` / `admin` / `sensitive_read` fail-closed；development 如需放行高风险也必须显式提供 policy/rule。
- `toolPermissionPolicyEngine` 在无 `PermissionManager` 时同样只允许 safe-read，拒绝高风险工具资源，避免生产缺省配置静默放行。
- hosted tool 风险分类已收口为 `ClassifyHostedToolResourceKind(...)`、`ClassifyHostedToolRiskTier(...)`、`ClassifyHostedToolPermissionRisk(...)`：file write=`RiskMutating`，code/shell=`RiskExecution`，MCP=`RiskNetworkExecution`，retrieval/file/web read=`RiskSafeRead`。
- `docs/architecture/权限控制系统详细设计-2026-04-24.md` 已补“当前职责边界（截至 2026-04-25）”，明确 `AuthorizationService`、`PolicyEngine`、`PermissionManager`、`ApprovalBackend`、`toolApprovalHandler`、`HITL InterruptManager`、`AuditSink`、hosted tool `ToolManager` 的职责和非职责。

## 6. 推荐实施顺序

推荐按以下顺序推进：

1. P1 公开入口与 public surface 收口。
2. P2 服务端多 Agent 执行链收口。
3. P5 授权与工具执行链硬化。
4. P7 Function calling 回归矩阵。
5. P3 Agent runtime 大文件拆分。
6. P4 统一运行时状态模型。
7. P6 SDK 级工具注册与示例产品化。
8. P8 可靠性、并发与取消。
9. P9 文档与架构守卫同步。

原因：

- P1 / P2 先收口入口，防止后续继续扩散双入口。
- P5 / P7 先保护工具执行安全和 provider 行为，直接影响可用性。
- P3 / P4 是中期工程治理，不应在入口未收口前大拆。
- P6 / P9 用来把内部能力转成外部可学习、可复用的产品表面。
