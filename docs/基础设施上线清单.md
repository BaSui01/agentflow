# 基础设施上线清单

> 文档类型：执行清单  
> 适用范围：`agentflow` 作为共享基础设施服务上线前的最小门槛  
> 当前状态：已完成

---

## 1. 目标

- [x] 让对外契约、默认配置、部署工件、运行时行为保持一致
- [x] 让服务在关键依赖缺失时表现为“明确失败”而不是“部分启动、语义不清”
- [x] 让运维团队可以据此完成部署、探针配置、发布和回滚

---

## 2. P0 必修

### 2.1 gRPC 对外契约统一

- [x] 统一 `gRPC` 暴露口径：当前执行口径为移除 `9090` 端口暴露、健康检查和部署文档中的 `gRPC` 描述
- [x] 同步修改默认配置、容器暴露端口、健康检查脚本、部署文档与 README
- [x] 通过相关测试与部署文档复验

完成定义：
- 运行时、默认配置、`Dockerfile`、`docker-compose.yml`、健康检查脚本不再出现“声明了 gRPC 但实际上未启动”的不一致
- 文档和代码的端口说明一致

建议验证：
- `go test ./cmd/agentflow ./config -count=1`
- `./scripts/arch_guard.ps1`

### 2.2 metrics/pprof 默认暴露收紧

- [x] 默认关闭 `pprof` 或仅允许显式开启
- [x] `metrics` 与 `pprof` 的监听方式改为安全默认值，避免直接无保护暴露到公网
- [x] 部署文档明确要求通过内网绑定、反向代理或等价保护方式访问观测端口

完成定义：
- 未显式开启时，`pprof` 不可访问
- `metrics`/`pprof` 暴露策略有清晰配置和文档

建议验证：
- `go test ./cmd/agentflow ./pkg/server ./pkg/middleware -count=1`

### 2.3 关键依赖启动门槛显式化

- [x] 明确哪些依赖缺失必须启动失败
- [x] 至少覆盖 `DB`、主 `LLM runtime`、工具审批存储、关键 Redis 依赖
- [x] 将“禁用部分能力继续启动”的路径压缩到非关键能力，避免上游误判服务健康

完成定义：
- 启动日志能明确区分“关键阻塞失败”和“可选能力降级”
- `/ready` 语义与启动门槛一致

建议验证：
- `go test ./cmd/agentflow ./internal/app/bootstrap ./api/... -count=1`

### 2.4 SQL 连接池接入主启动链

- [x] 将 `MaxOpenConns`、`MaxIdleConns`、`ConnMaxLifetime` 等配置真正接入 `serve` 启动链
- [x] 统一关闭流程，确保连接池和数据库连接一起优雅释放
- [x] 补充或更新对应测试

完成定义：
- 数据库连接池参数不是“仅存在于配置结构”，而是实际作用于服务启动
- 关闭流程不会遗留 SQL 连接

建议验证：
- `go test ./cmd/agentflow ./internal/app/bootstrap ./pkg/database -count=1`

### 2.5 探针语义与部署配置统一

- [x] 写清 `/health`、`/healthz`、`/ready`、`/readyz` 的职责边界
- [x] 部署文档和容器健康检查统一使用正确探针
- [x] 把关键依赖的 readiness 行为写入运维说明

完成定义：
- 活跃度与就绪度探针不再混用
- 容器健康检查、Kubernetes 探针建议、服务真实行为一致

建议验证：
- `go test ./api/handlers ./cmd/agentflow -count=1`

### 2.6 生产鉴权约束固化

- [x] 生产环境禁止以 `allow_no_auth=true` 的方式裸奔
- [x] 认证缺失时保持 fail-closed，并在部署文档中明确要求配置 `JWT` 或 `API Key`
- [x] 给受保护接口补一条最小 smoke test 说明

完成定义：
- 生产安全默认值清晰
- 无认证配置时不会误暴露业务接口

建议验证：
- `go test ./internal/app/bootstrap ./pkg/middleware ./config -count=1`

### 2.7 生产部署说明补齐

- [x] 输出生产部署所需的最小环境变量、依赖矩阵、启动顺序、端口与网络边界
- [x] 明确回滚、热重载边界和探针配置
- [x] 将本文档纳入部署文档导航

完成定义：
- 基础设施团队无需额外读代码也能完成首轮部署
