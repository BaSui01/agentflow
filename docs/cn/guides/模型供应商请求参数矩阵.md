# 模型供应商请求参数矩阵

> 更新时间：2026-05-02  
> 范围：主流文本 / Agent 调用参数，不替代多模态模型总表。  
> 目标：把各供应商官方请求参数归一到 AgentFlow 的 `types.AgentConfig{Model, Control, Tools}` 主面，并说明 provider 边界应如何翻译。  
> 口径：优先官方 API 文档；供应商模型 ID、provider code、JSON 字段名保持英文。

---

## 1. 统一落点总览

AgentFlow 不应在 `agent/runtime` 暴露各厂商原始 SDK request struct。统一落点如下：

| 语义 | AgentFlow 主面 | 低层 DTO | Provider 边界 |
|---|---|---|---|
| 模型与路由 | `Model.Provider` / `Model.Model` / `Model.RoutePolicy` | `ChatRequest.Metadata[chat_provider]` / `ChatRequest.Model` | native provider 或 `openaicompat` 选择 endpoint / fallback model |
| 输出长度 | `Model.MaxTokens` / `Model.MaxCompletionTokens` | `ChatRequest.MaxTokens` / `ChatRequest.MaxCompletionTokens` | OpenAI Responses 转 `max_output_tokens`；OpenAI-compatible 转 `max_tokens` / `max_completion_tokens`；Anthropic 转 `max_tokens`；Gemini 转 `generationConfig.maxOutputTokens` |
| 采样 | `Model.Temperature` / `Model.TopP` / penalties / `Stop` / `N` / logprobs | 同名 `ChatRequest` 字段 | provider 校验“不建议同时改 temperature 与 top_p”、不支持字段本地拒绝或忽略 |
| 推理 / thinking | `Model.ReasoningEffort` / `ReasoningSummary` / `ReasoningDisplay` / `ReasoningMode` / `ThinkingType` / `ThinkingLevel` / `ThinkingBudget` / `IncludeThoughts` | 同名 `ChatRequest` 字段 | OpenAI 转 `reasoning`；Anthropic 转 `thinking` / beta header；Gemini 转 `thinkingConfig`；compat provider 通过 hook 调整模型或字段 |
| 结构化输出 | `Model.ResponseFormat` | `ChatRequest.ResponseFormat` | OpenAI / compat 转 `response_format`；Gemini 转 `responseMimeType` / schema；Anthropic 保持 provider-native schema/tool 策略 |
| 工具调用 | `Tools.ToolChoice` / `Tools.ParallelToolCalls` / `Tools.ToolCallMode` / `Tools.AllowedTools` | `ChatRequest.Tools` / `ToolChoice` / `ParallelToolCalls` | OpenAI Responses 支持 allowed tools；Anthropic / Gemini / compat 各自转换 native tools 或 XML fallback |
| 连续对话 / 状态 | `Model.PreviousResponseID` / `ConversationID` / `ThoughtSignatures` / `CachedContent` | 同名 `ChatRequest` 字段 | OpenAI Responses 转 `previous_response_id` / `conversation`；Gemini 转 `cachedContent` / thought signature；Anthropic round-trip thinking blocks |
| 缓存 | `Model.PromptCacheKey` / `PromptCacheRetention` / `CacheControl` / `CachedContent` | 同名 `ChatRequest` 字段 | OpenAI prompt cache、Anthropic cache_control、Gemini cachedContent 分别在 provider 边界翻译 |
| 内置搜索 / 托管工具 | `Model.WebSearchOptions` / `Tools.ToolChoice` | `ChatRequest.WebSearchOptions` / `Tools` | OpenAI Responses web_search；Anthropic server tools；项目 Hosted Tools 仍走统一授权链 |

## 3. 对 Agent 框架的实施规则

1. **新增字段先问“是否跨供应商有统一语义”**：有统一语义时进 `types.ModelOptions` / `ToolProtocolOptions`；只有厂商私有语义时优先放 provider validation / catalog metadata。
2. **provider 边界负责协议翻译**：OpenAI 的 `max_output_tokens`、Gemini 的 `generationConfig.maxOutputTokens`、Anthropic 的 `max_tokens` 不应上抬成三个上层字段。
3. **推理字段要按产品语义分层**：
   - OpenAI：`ReasoningEffort` / `ReasoningSummary` / `Verbosity`。
   - Anthropic：`ThinkingType` / `ReasoningDisplay` / opaque thinking block round-trip。
   - Gemini：`ThinkingBudget` / `ThinkingLevel` / `IncludeThoughts`。
   - Compat 厂商：优先在 `llm/providers/vendor/chat_profiles.go` 用 hook / validation 表达差异。
4. **模型目录事实不等于请求参数**：模型 ID、上下文窗口、最大输出、stage、退役时间、capabilities 应进入 `types.ModelDescriptor` / `ModelCatalog` 或文档快照，不要散落在 provider hook 里。
5. **工具调用必须继续走统一授权链**：无论 native tools、OpenAI Responses hosted tools、Anthropic server tools，真正执行本地/业务工具仍要经过 `internal/usecase/authorization_service.go`。


### 3.1 模型目录快照加载

启动配置支持可选外部模型目录快照：

```yaml
llm:
  model_catalog_path: ./configs/model_catalog.json
```

也可以通过环境变量 `AGENTFLOW_LLM_MODEL_CATALOG_PATH` 覆盖。为空时 `internal/app/bootstrap.BuildModelCatalog` 使用 `types.DefaultModelCatalog()` 内置快照；非空时调用 `types.LoadModelCatalogFile` 加载 JSON，JSON 顶层可以是 `[]ModelDescriptor`，也可以是 `{ "models": [...] }`。启动组合会把 catalog 写入 `ServeHandlerSet.ModelCatalog` / server text runtime，并注入 `agent/runtime.CachingResolver.WithDefaultProvider` / `WithModelCatalog`；resolver 创建默认 agent 时会用 provider + alias 查 catalog，把配置别名规范化为 canonical model id。`types.ValidateModelCapabilities` 还会根据 tools、structured output、reasoning/thinking、web search 请求推导必需 capability，并在 `agent/runtime.Builder` / `RegisterDefaultRuntimeAgentFactory` 边界拒绝 catalog 已知但能力不匹配的模型。这条链路把“最新模型事实”限制在 catalog / bootstrap / resolver / builder 边界，避免 provider hook 继续散落硬编码模型 ID。
