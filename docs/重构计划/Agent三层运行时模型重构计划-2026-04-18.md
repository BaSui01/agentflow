# Agent 三层运行时模型重构计划（2026-04-18）

> 文档类型：可执行重构计划  
> 适用范围：`types/`、`agent/`、`llm/`、`agent/runtime/`、相关测试与文档  
> 核心约束：单轨替换；不保留兼容分支；不保留双实现；`ChatRequest` 仅作为 provider adapter DTO；运行时统一收口为“模型参数 + Agent 控制参数 + 工具调用协议”三层  
> 迁移策略：先收口执行选项与请求适配层，再推进 loop / reasoning / tool protocol 解耦，最后清理旧入口与文档守卫

## 1. 重构目标

### 1.1 运行时目标

- [ ] `AgentConfig` 的正式运行时主面收口到 `Model / Control / Tools`
- [ ] `RunOverrides` 采用分层 override，不再扩散平铺字段与 metadata 暗传
- [ ] `ResolvedExecutionOptions` 成为执行链唯一输入

### 1.2 协议目标

- [ ] `ToolChoice` 升级为强类型，替代 Agent 层 `any`
- [ ] `ToolProtocolRuntime` 统一 local / hosted / handoff / remote tool 的执行与结果回填
- [ ] `RemoteToolTransport` 统一 HTTP / MCP / A2A / stdio 远程工具承载

### 1.3 结构目标

- [ ] `ChatRequest` 只由 adapter 层构造
- [ ] `LoopExecutor` 从 `agent/integration.go` 拆出，并只消费 resolved control options
- [ ] `runtime.Builder` 只负责装配，不承担运行时语义分支

## 2. 现状问题（证据）

- [ ] 运行时配置分散在 `types.AgentConfig`、`agent.RunConfig`、`BuildOptions`、`Input.Context` 四层（证据：`types/config.go`、`agent/request.go`、`agent/runtime/builder.go`）
- [ ] `types.ChatRequest` 已承载大量模型与工具字段，但 Agent 上层未完整暴露（证据：`types/llm_contract.go`）
- [ ] `LoopExecutor`、reasoning 选择、validation、checkpoint/resume 逻辑集中在 `agent/integration.go`（证据：`agent/integration.go`）
- [ ] remote tool 仍分散在 hosted / MCP / A2A 多条线，缺少统一 transport 抽象（证据：`agent/hosted/`、`agent/protocol/mcp/`、`agent/protocol/a2a/`）
