- **已有提案**: [ADR-008](file:///d:/code/agentflow/docs/architecture/ADRs/008-agent-runtime-package-split.md)（状态：提议中）

  | `runtime/execution` | 执行引擎 |
  | `runtime/registry` | 注册中心 |
  | `runtime/adapters` | 适配器 |
- **影响范围**: 57 个外部导入文件、sdk/、api/handlers/、cmd/agentflow/、agent/team/、workflow/steps/

### P1-1: `api/handlers` 直接导入 `llm/core`

- **证据**: 12 个文件在 `api/handlers` 中直接 `import llm "github.com/BaSui01/agentflow/llm/core"`：
  - [chat_converter_helpers.go:8](file:///d:/code/agentflow/api/handlers/chat_converter_helpers.go#L8)
  - [chat_converter.go:8](file:///d:/code/agentflow/api/handlers/chat_converter.go#L8)
  - [chat_openai_request.go:9](file:///d:/code/agentflow/api/handlers/chat_openai_request.go#L9)
  - [apikey.go:10](file:///d:/code/agentflow/api/handlers/apikey.go#L10)
  - 以及 8 个测试文件
- **分析**: 适配层不应直接依赖业务层的核心抽象（`llm/core`）。虽然 `llm/core` 本身作为 Layer 1 被允许作为适配层下游，但由于 handler 层已有 `internal/usecase` 作为协议转换层，应通过 usecase 间接引用或走 `types/` 中的共享接口
- **严重程度**: P1 —— 不阻塞编译，但削弱分层隔离性
