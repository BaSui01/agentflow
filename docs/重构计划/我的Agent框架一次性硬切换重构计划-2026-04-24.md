## 2. 现状问题（证据）

- [x] 问题 1：公开表面过多，调用方需要理解 `sdk`、`agent/execution/runtime`、`agent/collaboration/team`、`multiagent.GlobalModeRegistry` 等多层语义（证据：`sdk/runtime.go`、`internal/usecase/agent_service.go:227`、`docs/architecture/AgentFlow收口改造方案与实施清单-2026-04-23.md`）
- [x] 问题 2：单 Agent runtime 与显式 workflow 编排边界不够硬，`parallel / loop / orchestration` 语义同时散布在 `multiagent` 和 `workflow`（证据：`agent/collaboration/multiagent/default_modes.go`、`workflow/steps/orchestration.go`）
- [x] 问题 3：权限能力已经存在，但仍停留在“工具审批系统”，没有上升到统一 `AuthN/AuthZ/Approval/Audit` 模型（证据：`internal/app/bootstrap/agent_tool_approval_builder.go`、`internal/app/bootstrap/agent_tool_policy_builder.go`、`docs/architecture/权限控制系统详细设计-2026-04-24.md`）
- [x] 问题 4：runtime 仍有大文件和混合职责，长期演进风险高（证据：`agent/execution/runtime/request_runtime.go`、`agent/execution/runtime/agent_builder.go`、`agent/execution/runtime/interfaces_runtime.go`）
- [x] 问题 5：当前目录结构更多体现历史演进，不够体现“自己的框架产品面”（证据：`agent/execution/runtime`、`agent/collaboration/*`、`agent/adapters/teamadapter` 的并存）

## 3. 目标架构

### 3.1 唯一正式入口

- [x] 唯一仓库级正式入口：`sdk.New(opts).Build(ctx)`
- [x] 唯一单 Agent 正式入口：`agent/runtime`
- [x] 唯一多 Agent 正式入口：`agent/team`
- [x] 唯一显式编排入口：`workflow/runtime`
- [x] 唯一统一授权入口：`internal/usecase/authorization_service.go`
- [x] 旧正式入口降级为内部构件或直接删除：`agent/execution/runtime`、`agent/collaboration/team`、`agent/collaboration/multiagent`、`agent/adapters/teamadapter`

### 3.2 目标分层 / 分包

- [x] `sdk/`：仓库级统一装配入口
- [x] `agent/runtime/`：单 Agent runtime（Agent/Tool/Handoff/Session/RunConfig/RunEvent）
- [x] `agent/team/`：team facade + team contract + internal engines
- [x] `workflow/runtime/`：显式流程编排（Sequential / Parallel / Branch / Loop / Subgraph / HITL / Checkpoint）
- [x] `types/authz.go`：零依赖权限契约
- [x] `internal/usecase/authorization_service.go`：统一授权服务

### 3.3 依赖方向与主链约束

- [x] 不破坏现有层级依赖方向
- [x] 不绕过 `cmd -> bootstrap -> api -> domain` 启动主链
- [x] 不在 handler / adapter 层回灌业务实现
- [x] `workflow` 只依赖 `agent` contract，不吞并 `agent` runtime

### 3.4 根目录治理目标（如适用）

- [x] 顶层目录分类方案已定义（产品源码 / 正式文档 / 构建治理 / 示例验证 / 状态产物 / 外部参考）
- [x] `docs/architecture/` 作为活跃契约目录，`docs/archive/` 作为历史归档目录

### 3.5 外部参考框架对标（本轮硬切换的设计基线）

- [x] OpenAI Agents SDK：保留最小原语（Agent / Tool / Handoff / Session / Trace）
- [x] Google ADK：借鉴 workflow agents（Sequential / Parallel / Loop）和 Session / State / Memory
- [x] Microsoft Agent Framework：借鉴 Agent 与 Workflow 的硬边界
- [x] LangGraph：借鉴 durable execution / checkpoint / HITL / graph orchestration
- [x] PydanticAI：借鉴 typed output / deps / durable execution
- [x] CrewAI：借鉴 Team 与 Flow 的职责分离
- [x] Mastra：借鉴 stateful agents / workflows / MCP
- [x] Pi：借鉴 coding-agent harness / session / RPC/headless 模式
