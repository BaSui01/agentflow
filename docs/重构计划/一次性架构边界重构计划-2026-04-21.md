## 1. 重构目标

### 1.1 业务目标

- [ ] API Handler 仅负责协议解析、参数校验、错误映射、响应序列化，不再承载业务执行与运行时装配。
- [ ] 所有 HTTP 业务执行入口统一下沉到 `internal/usecase/`，避免 `api/handlers` 内继续扩散 service/usecase。
- [x] 服务启动时只保留一个 handler 总装配入口，不再让 `cmd/agentflow/server_handlers_runtime.go` 承担大段条件编排。

### 1.2 架构目标

- [x] `internal/usecase/` 不再依赖 `api/` DTO；跨层输入输出改为 usecase 自有 DTO 或复用 `types/`。
- [x] `api/handlers` 每个 handler 只保留一个主构造入口：`NewXxxHandler(service, logger)`。
- [ ] 热更新只替换 usecase/runtime 持有器，不再更新 handler 内部 runtime。
- [ ] `agent/` 根包从“多职责大文件”收敛为“稳定门面 + 组装入口”，大实现下沉到子目录。

## 2. 现状问题（证据）

- [x] `api/handlers/multimodal.go` 同时包含 config 驱动 provider 构造、gateway/router 装配、prompt enhancer/optimizer 初始化，已越过适配层边界（证据：`api/handlers/multimodal.go:43-90`, `115-277`）。
- [x] `api/handlers` 目录内仍存在 7 个 service 文件，说明适配层和应用层混放（证据：`api/handlers/apikey_service.go`, `multimodal_service.go`, `rag_service.go`, `tool_approval_service.go`, `tool_provider_service.go`, `tool_registry_service.go`, `workflow_service.go`）。
- [x] `internal/usecase/chat_service.go` 直接依赖 `api.ChatRequest` / `api.ChatResponse`，应用层反向绑定 transport DTO（证据：`internal/usecase/chat_service.go:8-26`）。
- [x] `cmd/agentflow/server_handlers_runtime.go` 已显式标注 `TODO(refactor)`，且 `initHandlers()` 仍包含大量条件装配与运行时拼接（证据：`cmd/agentflow/server_handlers_runtime.go:17-18`, `44-323`）。
- [x] `internal/app/bootstrap/` 生产文件 22 个，其中 builder 文件 18 个，说明 builder 已碎片化但还缺“唯一总装配入口”（证据：目录清点 `internal/app/bootstrap/*`）。

- [x] 当前文档要求 `api` 只做协议转换、`cmd` 只做装配、`bootstrap` 只做启动期构建；现状与目标边界存在偏差（证据：`AGENTS.md:16-25`, `docs/architecture/startup-composition.md:11-15`）。
