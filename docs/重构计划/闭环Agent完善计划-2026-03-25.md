## 1. 重构目标

### 1.1 业务目标

- [ ] 默认 `BaseAgent.Execute(...)` 升级为“感知 -> 分析 -> 规划 -> 执行 -> 观察 -> 评估 -> 决策下一步”的闭环主链
- [ ] Agent 能基于统一完成判定持续迭代，直到“已解决 / 达到边界 / 需要人工介入”之一
- [ ] 高级推理模式（`react/rewoo/plan_and_execute/dynamic_planner/tree_of_thought/reflection`）收敛为统一策略选择，而不是散落式直调
- [ ] 默认 HTTP / runtime 构造出的 agent 直接具备闭环执行能力，不需要额外手工拼装

### 1.2 架构目标

- [ ] 新增统一闭环状态对象 `LoopState`，禁止在多处散落维护“消息/计划/观察/轮次/停止原因”
- [ ] 新增统一闭环执行器 `LoopExecutor`，`Execute(...)` 仅保留一个默认入口
- [ ] 新增完成判定抽象 `CompletionJudge`，停止条件不再只依赖 `MaxReActIterations/timeout/retry`
- [ ] `Plan(...)` 与 `Observe(...)` 从“独立辅助 API”升级为默认主链阶段
- [ ] 删除或下沉默认主链之外的重复循环控制逻辑，避免出现多套“直到解决”为止的分叉实现

## 2. 现状问题（证据）

- [x] 默认主链仍是 `Execute -> executeWithPipeline -> executeCore -> ChatCompletion/ReAct`，没有统一“规划/观察/再规划”闭环（证据：`agent/react.go`、`agent/integration.go`、`agent/completion.go`）
- [x] `Plan(...)` 是独立方法，不会被默认 `Execute(...)` 自动串联（证据：`agent/react.go`）
- [x] `Observe(...)` 只承担反馈写入，不是循环中的统一 Observe 阶段（证据：`agent/react.go`）
- [x] `builder.WithReasoning(...)` 已持有 `reasoningRegistry`，但默认构建出的 agent 主链未消费该 registry（证据：`agent/builder.go`）
- [x] `reasoning/` 已实现 `ReWOO/PlanAndExecute/DynamicPlanner/TreeOfThought/Reflexion`，但这些模式当前不是默认主链的一部分（证据：`agent/reasoning/*.go`）
- [x] 当前停止条件主要是 `MaxReActIterations`、guardrails retry、timeout、reflection 最大轮次，不是统一 solved 判定（证据：`agent/request.go`、`agent/completion.go`、`agent/reflection.go`）

## 3. 目标架构（重构后唯一形态）

### 3.1 默认执行状态机

- [ ] `Perceive`：汇总输入、上下文、记忆、历史消息、工具观察结果
- [ ] `Analyze`：任务复杂度评估、模式选择、风险识别
- [ ] `Plan`：生成或更新当前计划；简单任务允许生成单步计划而不是跳过计划阶段
- [ ] `Act`：执行当前步骤（LLM 推理 / 工具调用 / 子代理执行）
- [ ] `Observe`：将执行结果、工具结果、外部反馈统一写入观察集合
- [ ] `Evaluate`：统一判定是否已解决、是否低质量、是否需要继续
- [ ] `DecideNext`：只允许 `done / continue / replan / reflect / escalate`

### 3.2 单一闭环运行对象

- [ ] `LoopState` 统一承载 `Goal/Plan/Observations/Attempts/Messages/Iteration/StopReason/Confidence`
- [ ] `LoopExecutor` 统一驱动阶段流转，不允许多个模块各自维护主循环
- [ ] `CompletionJudge` 统一输出 `Solved/NeedReplan/NeedReflection/NeedHuman/Confidence/Reason`

### 3.3 推理模式统一调度

- [ ] `react` 作为低复杂度默认执行模式
- [ ] `rewoo` 作为工具密集型模式
- [ ] `plan_and_execute` 作为可重规划模式
- [ ] `dynamic_planner` 作为回溯/替代路径模式
- [ ] `tree_of_thought` 作为高不确定性多分支模式
- [ ] `reflection` 作为闭环中的质量修正阶段，不再只是外层装饰器

---

## 4. 模式边界与归属

### 4.1 默认单 Agent 主链

- [ ] `BaseAgent.Execute(...)` 仅负责调用 `LoopExecutor`
- [ ] `LoopExecutor` 仅负责任务状态机流转、统一停止语义、阶段切换、状态持久化
- [ ] `executeCore(...)` 降级为单步执行原语，不再代表完整 Agent 生命周期

### 4.2 Reasoning 策略层

- [ ] `agent/reasoning/` 仅提供可插拔求解策略，不直接拥有默认主链
- [ ] `ReWOO/PlanAndExecute/DynamicPlanner/TreeOfThought/Reflexion` 统一经 `ReasoningModeSelector` 调度
- [ ] 禁止调用方绕过默认主链直接把某个 reasoning 模式当作新的默认执行入口

### 4.3 Planner 基础设施层

- [ ] `agent/planner/` 仅负责任务图、任务依赖、任务分发与执行聚合
- [ ] `planner/` 不直接替代 `LoopExecutor`，只作为 `Plan` / `Act` 阶段内部能力
- [ ] 禁止形成“默认主链一套计划执行 + planner 另一套计划执行”双轨

### 4.4 Multi-Agent 协作层

- [ ] `multiagent/` 只负责多 Agent 协作，不拥有默认单 Agent 主链
- [ ] `ModeLoop` 仅保留为多 Agent 编排模式，不得演变为第二套默认闭环执行器
- [ ] 单 Agent 默认闭环与多 Agent loop 模式必须共享统一停止语义与可观测字段

---

## 5. 统一停止语义

### 5.1 停止原因枚举（唯一口径）

- [ ] `solved`：完成判定器确认任务已解决
- [ ] `max_iterations`：达到闭环总轮次上限
- [ ] `timeout`：达到整体执行超时
- [ ] `blocked`：无法继续规划或执行，且无可用替代路径
- [ ] `need_human`：需要人工确认、补充信息或批准
- [ ] `validation_failed`：输出/输入校验无法恢复
- [ ] `tool_failure_unrecoverable`：工具故障且重试/替代失败

### 5.2 旧停止条件迁移规则

- [ ] `MaxReActIterations` 迁移为单步执行预算，不再等价于“任务完成”
- [ ] `Reflection.MaxIterations` 迁移为质量修正预算，不再作为顶层 stop reason
- [ ] `Reflexion.MaxTrials` 迁移为策略内部尝试预算
- [ ] `PlanAndExecute.MaxReplanAttempts` 迁移为规划预算
- [ ] `DynamicPlanner.MaxPlanDepth/ConfidenceThreshold` 迁移为策略内部边界，不替代统一完成判定

### 5.3 完成判定输出

- [ ] `CompletionJudge` 输出 `Solved/Confidence/Reason/NeedReplan/NeedReflection/NeedHuman/StopReason`
- [ ] 所有默认主链停止都必须经过 `CompletionJudge` 或显式异常中止
- [ ] API / 持久化 / 日志统一使用同一 `stop_reason`
