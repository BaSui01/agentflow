## 1. 重构目标

### 1.1 业务目标

- [x] 默认 `BaseAgent.Execute(...)` 升级为“感知 -> 分析 -> 规划 -> 执行 -> 观察 -> 评估 -> 决策下一步”的闭环主链
- [x] Agent 能基于统一完成判定持续迭代，直到“已解决 / 达到边界 / 需要人工介入”之一
- [x] 高级推理模式（`react/rewoo/plan_and_execute/dynamic_planner/tree_of_thought/reflection`）收敛为统一策略选择，而不是散落式直调
- [x] 默认 HTTP / runtime 构造出的 agent 直接具备闭环执行能力，不需要额外手工拼装

### 1.2 架构目标

- [x] 新增统一闭环状态对象 `LoopState`，禁止在多处散落维护“消息/计划/观察/轮次/停止原因”
- [x] 新增统一闭环执行器 `LoopExecutor`，`Execute(...)` 仅保留一个默认入口
- [x] 新增完成判定抽象 `CompletionJudge`，停止条件不再只依赖 `MaxReActIterations/timeout/retry`
- [x] `Plan(...)` 与 `Observe(...)` 从“独立辅助 API”升级为默认主链阶段
- [x] 删除或下沉默认主链之外的重复循环控制逻辑，避免出现多套“直到解决”为止的分叉实现

## 2. 现状问题（证据）

- [x] 默认主链仍是 `Execute -> executeWithPipeline -> executeCore -> ChatCompletion/ReAct`，没有统一“规划/观察/再规划”闭环（证据：`agent/react.go`、`agent/integration.go`、`agent/completion.go`）
- [x] `Plan(...)` 是独立方法，不会被默认 `Execute(...)` 自动串联（证据：`agent/react.go`）
- [x] `Observe(...)` 只承担反馈写入，不是循环中的统一 Observe 阶段（证据：`agent/react.go`）
- [x] `builder.WithReasoning(...)` 已持有 `reasoningRegistry`，但默认构建出的 agent 主链未消费该 registry（证据：`agent/builder.go`）
- [x] `reasoning/` 已实现 `ReWOO/PlanAndExecute/DynamicPlanner/TreeOfThought/Reflexion`，但这些模式当前不是默认主链的一部分（证据：`agent/reasoning/*.go`）
- [x] 当前停止条件主要是 `MaxReActIterations`、guardrails retry、timeout、reflection 最大轮次，不是统一 solved 判定（证据：`agent/request.go`、`agent/completion.go`、`agent/reflection.go`）

## 3. 目标架构（重构后唯一形态）

### 3.1 默认执行状态机

- [x] `Perceive`：汇总输入、上下文、记忆、历史消息、工具观察结果
- [x] `Analyze`：任务复杂度评估、模式选择、风险识别
- [x] `Plan`：生成或更新当前计划；简单任务允许生成单步计划而不是跳过计划阶段
- [x] `Act`：执行当前步骤（LLM 推理 / 工具调用 / 子代理执行）
- [x] `Observe`：将执行结果、工具结果、外部反馈统一写入观察集合
- [x] `Evaluate`：统一判定是否已解决、是否低质量、是否需要继续
- [x] `DecideNext`：只允许 `done / continue / replan / reflect / escalate`

### 3.2 单一闭环运行对象

- [x] `LoopState` 统一承载 `Goal/Plan/Observations/Attempts/Messages/Iteration/StopReason/Confidence`
- [x] `LoopExecutor` 统一驱动阶段流转，不允许多个模块各自维护主循环
- [x] `CompletionJudge` 统一输出 `Solved/NeedReplan/NeedReflection/NeedHuman/Confidence/Reason`

### 3.3 推理模式统一调度

- [x] `react` 作为低复杂度默认执行模式
- [x] `rewoo` 作为工具密集型模式
- [x] `plan_and_execute` 作为可重规划模式
- [x] `dynamic_planner` 作为回溯/替代路径模式
- [x] `tree_of_thought` 作为高不确定性多分支模式
- [x] `reflection` 作为闭环中的质量修正阶段，不再只是外层装饰器

## 4. 模式边界与归属

### 4.1 默认单 Agent 主链

- [x] `BaseAgent.Execute(...)` 仅负责调用 `LoopExecutor`
- [x] `LoopExecutor` 仅负责任务状态机流转、统一停止语义、阶段切换、状态持久化
- [x] `executeCore(...)` 降级为单步执行原语，不再代表完整 Agent 生命周期

### 4.2 Reasoning 策略层

- [x] `agent/reasoning/` 仅提供可插拔求解策略，不直接拥有默认主链
- [x] `ReWOO/PlanAndExecute/DynamicPlanner/TreeOfThought/Reflexion` 统一经 `ReasoningModeSelector` 调度
- [x] 禁止调用方绕过默认主链直接把某个 reasoning 模式当作新的默认执行入口

### 4.3 Planner 基础设施层

- [x] `agent/planner/` 仅负责任务图、任务依赖、任务分发与执行聚合
- [x] `planner/` 不直接替代 `LoopExecutor`，只作为 `Plan` / `Act` 阶段内部能力
- [x] 禁止形成“默认主链一套计划执行 + planner 另一套计划执行”双轨

### 4.4 Multi-Agent 协作层

- [x] `multiagent/` 只负责多 Agent 协作，不拥有默认单 Agent 主链
- [x] `ModeLoop` 仅保留为多 Agent 编排模式，不得演变为第二套默认闭环执行器
- [x] 单 Agent 默认闭环与多 Agent loop 模式必须共享统一停止语义与可观测字段

---

## 5. 统一停止语义

### 5.1 停止原因枚举（唯一口径）

- [x] `solved`：完成判定器确认任务已解决
- [x] `max_iterations`：达到闭环总轮次上限
- [x] `timeout`：达到整体执行超时
- [x] `blocked`：无法继续规划或执行，且无可用替代路径
- [x] `need_human`：需要人工确认、补充信息或批准
- [x] `validation_failed`：输出/输入校验无法恢复
- [x] `tool_failure_unrecoverable`：工具故障且重试/替代失败

### 5.2 旧停止条件迁移规则

- [x] `MaxReActIterations` 迁移为单步执行预算，不再等价于“任务完成”
- [x] `Reflection.MaxIterations` 迁移为质量修正预算，不再作为顶层 stop reason
- [x] `Reflexion.MaxTrials` 迁移为策略内部尝试预算
- [x] `PlanAndExecute.MaxReplanAttempts` 迁移为规划预算
- [x] `DynamicPlanner.MaxPlanDepth/ConfidenceThreshold` 迁移为策略内部边界，不替代统一完成判定

### 5.3 完成判定输出

- [x] `CompletionJudge` 输出 `Solved/Confidence/Reason/NeedReplan/NeedReflection/NeedHuman/StopReason`
- [x] 所有默认主链停止都必须经过 `CompletionJudge` 或显式异常中止
- [x] API / 持久化 / 日志统一使用同一 `stop_reason`

## 13. 现状基线与并行分析结论

### 13.1 默认单 Agent 当前真实执行链

- [x] HTTP 入口当前为 `api/handlers/agent.go -> internal/usecase/agent_service.go -> agent.Agent.Execute(...)`
- [x] 默认单 Agent 执行链当前为 `agent/react.go:Execute -> agent/integration.go:executeWithPipeline -> agent/integration.go:coreExecutor -> agent/react.go:executeCore -> agent/completion.go:ChatCompletion`
- [x] `Plan(...)` 当前仅通过 `HandlePlanAgent -> PlanAgent -> agent/react.go:Plan` 独立调用，不会被默认 `Execute(...)` 自动串联
- [x] `Observe(...)` 当前只在 `agent/react.go` 独立实现，默认生产主链不会在 `Execute(...)` 后自动进入观察阶段

### 13.2 默认主链当前停机/循环缺口

- [x] Agent 顶层没有统一 `Plan -> Act -> Observe -> Evaluate -> DecideNext` 闭环，顶层 `Execute(...)` 仅做一次主执行委托
- [x] 真正的循环主要仍在 `llm/capabilities/tools/react.go` 的 `ReActExecutor.Execute/ExecuteStream` 中，以 `MaxIterations` 和工具调用结果为主
- [x] `executeCore(...)` 仅内置 output guardrails retry，不承担统一 solved 判定
- [x] `multiagent.ModeLoop` 当前仍依赖 `stopKeyword` 文本哨兵，不可直接复用为默认单 Agent 停止语义

### 13.3 Reasoning 模式接入缺口

- [x] `builder.WithReasoning(...)` 与 `agent/runtime/builder.go` 已支持 `ReasoningRegistry` 注入
- [x] 默认 runtime 工厂当前未显式注入 `ReasoningRegistry`
- [x] 默认主链 `Execute -> executeWithPipeline -> coreExecutor -> executeCore` 当前没有任何位置读取 `reasoning.PatternRegistry`
- [x] `ReasoningModeSelector` 最适合挂在 `agent/integration.go:coreExecutor` 附近，而不是放到 `agent/reasoning/`、`workflow/`、`api/` 或 `cmd/`

### 13.4 LoopState / CompletionJudge / 恢复建议落点

- [x] `LoopState` 应优先复用 `agent/checkpoint_stores.go` 中的 `Checkpoint / ExecutionContext`，不应复用 workflow DAG 专用 `EnhancedCheckpoint`
- [x] 单 Agent 恢复路径应继续挂在 `BaseAgent.Execute(...)` 主链，多 Agent loop 恢复则继续挂在 `ModeRegistry.Execute -> loopModeStrategy.Execute`
- [x] `CompletionJudge` 第一批优先收敛 `executeCore(...)`、`ReflectionExecutor.ExecuteWithReflection(...)`、`ReflexionExecutor.Execute(...)`、`PlanAndExecute.Execute(...)`、`DynamicPlanner.shouldContinue(...)`
- [x] 第一批统一 `stop_reason` 只先覆盖 `solved/max_iterations/timeout/validation_failed/tool_failure_unrecoverable/blocked`

### 13.5 API / 流式 / 观测字段改造面

- [x] 同步响应字段应沿 `agent.Output -> internal/usecase.AgentExecuteResponse -> api/handlers/agent.go` 抬升
- [x] 流式字段应沿 `agent/completion.go:RuntimeStreamEvent -> api/handlers/agent.go:HandleAgentStream` 抬升
- [x] `current_stage` 最适合先挂在 `RuntimeStreamEvent` 与 `BaseAgent.Transition(...)` 周边
- [x] `iteration_count` 最适合从 `chatCompletionStreaming` 消费底层 ReAct 迭代事件后抬升
- [x] `checkpoint_id/resumable` 最适合先挂在 `CheckpointManager.SaveCheckpoint/ResumeFromCheckpoint` 与 `executeCore(...)` 的输出汇总/log 点
