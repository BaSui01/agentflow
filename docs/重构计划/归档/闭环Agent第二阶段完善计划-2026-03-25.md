## 1. 重构目标

### 1.1 业务目标

- [x] 默认单 Agent 在循环过程中显式记录“未完成事项 / 验收标准 / 剩余风险”，而不是只保留摘要文本
- [x] `validate` 阶段升级为可插拔验收器链，而不是主要依赖零散 metadata
- [x] 代码任务、工具任务、通用任务至少各有一类默认验收策略，避免“阶段存在但验收空心化”

### 1.2 架构目标

- [x] `LoopState` 扩展 `AcceptanceCriteria / UnresolvedItems / RemainingRisks / ValidationSummary`
- [x] `Checkpoint / ExecutionContext` 持久化这些字段，恢复后继续走同一 `LoopExecutor`
- [x] `LoopValidator` 支持链式组合，默认主链只保留一个统一验收入口
- [x] `CompletionJudge` 消费结构化验证结果与未完成项，不再主要依赖 metadata 推断 solved

## 2. 现状问题（第二阶段）

- [x] 当前 `validate` 已进入主链，但默认实现仍偏向 metadata gate，缺少可插拔任务级验收器
- [x] `LoopState` 当前尚未显式记录未完成事项、剩余风险、验收总结
- [x] checkpoint 恢复虽已保留阶段/轮次，但还不能恢复结构化验收上下文
- [x] 对代码任务、工具任务的验收仍偏弱，尚未形成稳定默认策略

## 3. 执行计划

### Phase-1：结构化状态

- [x] 为 `LoopState` 增加验收标准、未完成事项、剩余风险、验证摘要字段
- [x] 补齐 `restoreFromContext / CheckpointVariables / PopulateCheckpoint` 对应映射
- [x] 增加 `loop_state_test.go` 针对新字段的恢复与持久化测试

### Phase-2：验收器链

- [x] 新增 `LoopValidationIssue` / `LoopValidationSummary` 等结构化结果
- [x] 将 `LoopValidator` 升级为链式执行，支持多个默认 validator 组合
- [x] 保持默认主链只走一个 `validator()` 入口，不引入并行旁路

### Phase-3：任务级默认验收器

- [x] 实现通用验收器：消费 acceptance criteria、未完成项、显式 validation 状态
- [x] 实现工具任务验收器：在工具密集/验证必需场景下补强 verification 语义
- [x] 实现代码任务验收器：对代码/修复类任务补充“测试/验证尚未完成”的默认判定

### Phase-4：判定收敛与验收

- [x] 让 `CompletionJudge` 以结构化验证结果为主，不再把 metadata 当唯一事实来源
- [x] 补齐 `completion_judge_test.go`、`loop_executor_test.go`、`integration_test.go`
- [x] 通过 `go test ./agent -run "LoopState|LoopExecutor|CompletionJudge|BaseAgent_Execute_DefaultClosedLoop" -count=1`
