# LLM 统一入口收口计划（2026-04-17）

> 文档类型：可执行重构计划  
> 适用范围：`workflow/`、`agent/reasoning/`、`agent/structured/`、`agent/evaluation/`、`agent/deliberation/` 以及相关启动装配/守卫/测试

> 并行策略：`WS-1 workflow`、`WS-2 reasoning`、`WS-3 structured/evaluation/deliberation` 可并行；`WS-4 守卫/验收` 串行收尾

## 1. 重构目标

### 1.1 业务目标

- [x] `workflow/agent/api` 业务代码不再直接调用 `llm.Provider.Completion/Stream`
- [x] 所有业务侧模型请求统一走 `llmcore.Gateway.Invoke/Stream`
- [x] 统一 token/cost/provider/model 统计口径，只认 `llm/gateway` 出口
- [x] 收口后不保留 provider 直调旧路径，不做双轨兼容

### 1.2 架构目标

- [x] `llm.Provider` 仅留在 `llm/providers/*`、`llm/runtime/*`、`internal/app/bootstrap/*` 等构造/适配边界
- [x] `workflow/` 若保留本地抽象，只允许以 `llmcore.Gateway` 为底座，不允许再包装裸 `llm.Provider`
- [x] `agent/reasoning`、`agent/structured`、`agent/evaluation`、`agent/deliberation` 对外构造签名统一改为接收 `llmcore.Gateway`
- [x] 默认运行时装配显式传递真实 gateway，而不是把 gateway 降格为 provider adapter 后再下传

### 1.3 允许的唯一入口

- [x] 统一入口定义为 [llm/core/contracts.go](/E:/code/agentflow/llm/core/contracts.go#L33) 的 `llmcore.Gateway`
- [x] 若业务子模块需要窄接口，只允许定义基于 `llmcore.Gateway` 的薄适配接口，不允许重新引入第二套 provider 入口
- [x] `Capability=chat` 的请求统一通过 `UnifiedRequest{Capability: CapabilityChat, Payload: *llm.ChatRequest}`
