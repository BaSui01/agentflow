# AgentFlow 框架优化执行计划

> 生成日期：2026-03-06
> 基于：Web 搜索（grok-search + context7）+ 代码静态分析

---

## 一、分析摘要

| 方向 | 当前状态 | 核心差距 |
|------|----------|----------|
| Agent×DAG 集成 | `AgentStep`+`OrchestrationStep` 已接入 DAG | 缺 LoopAgent 模式、Agent 子图、DSL agent_id 映射 |
| MCP 协议 | 自研客户端，仅支持 `ListTools`/`CallTool`+Stdio/SSE | 缺 Resources/Prompts/InProcess/StreamableHTTP/Auth/多服务端管理 |
| RAG 混合检索 | RRF+Weighted 融合、BM25 可调、规则+LLM 路由 | 语义缓存无 Prometheus；Graph RAG 无自动实体抽取；无 HyDE 查询重写 |
| 并发与性能 | LLM 工具层并行，Agent 主链路串行 | `hostedToolManager` 串行执行 tool_calls；无 pprof；无工具 Prometheus 指标 |
| 可观测性 | Prometheus HTTP/LLM/Agent/Cache 指标 + OTel trace | 无工具调用指标；成本跟踪缺 Session/Tool 维度；检查点无自动清理 |

## 二、P0 任务（立即执行）

### P0-1: Agent 主链路并行工具调用

**问题**：`hostedToolManager.ExecuteForAgent` 串行执行所有 tool_calls，LLM 返回多个并行工具调用时延迟叠加。

**目标**：改为 goroutine + errgroup 并行执行，延迟降为最慢单工具时间。

**文件**：`internal/app/bootstrap/agent_tooling_runtime_builder.go`

**改动点**（行 329-348）：
- [x] 将 `for i, call := range calls` 循环改为 goroutine 并行
- [x] 使用 `sync.WaitGroup` + `context`
- [x] 保留 `results[i]` 的索引对应关系
- [x] 单个工具失败不阻塞其他工具（与 `DefaultExecutor` 行为一致）

**验收**：
- [x] 编译通过
- [x] 已有测试不回归

**问题**：Agent 层缺少循环模式（对比 Google ADK `LoopAgent`），只有 DAG 层的 `NodeTypeLoop`。

**目标**：在 `agent/multiagent/` 新增 `loop` 模式策略。

**改动点**：

- [x] 配置：`max_iterations`（默认 5）、`stop_keyword`（默认 `LOOP_COMPLETE`）
- [x] 每轮将上一轮 Output 作为下一轮 Input
- [x] 注册到 `ModeRegistry` 默认模式中（name: `"loop"`）

**验收**：

- [x] 所有已有测试通过

## 六、风险与注意事项

1. **P0-1 并行化**：需确保工具执行的线程安全性，特别是 Shell/File 工具可能有副作用
2. **MCP 迁移**：如迁移到 `mark3labs/mcp-go`，需评估 API 兼容性和现有测试覆盖
3. **架构守卫**：所有改动须通过 `architecture_guard_test.go` 和 `scripts/arch_guard.ps1`
4. **禁止兼容代码**：按 AGENTS.md 规则，修改时必须删除旧实现，不允许新老并存
