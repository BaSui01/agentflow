## 1. 变更概要

2026-04-26 完成了一批聚焦于 API handlers 层、agent runtime 构建、启动装配链路和用例层的重构，主要目标是继续推进架构分层治理和职责收口。

---

## 2. 变更清单

### 2.1 API Handlers 重构与参数处理统一

- 重构 `api/handlers` 各 handler，统一参数处理模式
- Handler 层只做协议转换（request decode / validation / auth context / error mapping / response serialization / stream writing）
- 删除 handler 中的 `gorm` store 实现，owner store 下沉到 `llm/runtime/router/apikey_store_gorm.go` 与 `agent/integration/hosted/tool_provider_store_gorm.go`
- `chat.go` / `workflow.go` 不再直接依赖 `llm/core`、`workflow/core`

### 2.2 Agent Runtime 构建与生命周期结构重构

- 重构 `agent/runtime` 构建与生命周期结构
- `agent/runtime.Builder` 保持为单 Agent 正式构建入口
- 生命周期状态管理、信号量并发控制与配置锁分离的既有设计保留

### 2.3 启动装配链路重构 — Handler 与 Runtime 集合分离

- 重构 `internal/app/bootstrap` 启动装配，将 Handler 集合与 Runtime 集合显式分离
- `bootstrap` 提供 `BuildServeHandlerSet(...)` 作为唯一 serve-time 装配入口
- `bootstrap` 提供 `ApplyReloadedTextRuntimeBindings(...)` 作为唯一 text hot reload rebinding 入口
- `bootstrap` 提供 `BuildToolingHandlerBundle(...)` 作为 hosted tool / provider / approval handler bundle 入口
- `cmd/agentflow` 只保留生命周期、字段持有与调用 bootstrap 的职责，不再直接 new handler / service / workflow facade

### 2.4 用例层重构 — Agent 执行模式与权限服务分离

- 重构 `internal/usecase`，分离 Agent 执行模式和权限服务
- `internal/usecase/authorization_service.go` 保持为统一授权入口
- Agent 执行模式（AgentExecutionModes）独立提取
- usecase 为 handler-facing chat/workflow 提供 usecase-owned DTO、stream/event 契约与 runtime ref

### 2.5 工作流编排与检查点机制增强

- 增强工作流编排和检查点机制
- `workflow/runtime.Builder` 保持为显式编排正式入口

### 2.6 架构守卫与文档同步

- 更新架构守卫测试
- 同步重构计划文档
