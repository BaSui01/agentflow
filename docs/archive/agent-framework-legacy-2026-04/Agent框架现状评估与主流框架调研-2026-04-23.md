## 3. 主流 Agent 框架调研（官方资料，按 2026-04-24 可见状态）

> 说明：这里的“主流”不是做星数排行榜，也不可能穷尽网络上所有实验项目；本轮按**官方文档成熟度、生态影响力、能力覆盖面、与本项目问题空间的相关性**选取，并补充长尾框架的“可借鉴能力矩阵”。

### 3.1 对比总表

| 框架 | 主要定位 | 强项 | 代价 / 风险 | 对本项目最值得借鉴的点 |
|---|---|---|---|---|
| OpenAI Agents SDK | 轻量运行时 + handoff + tracing | 原语少、执行环清晰、事件流、sandbox、MCP、guardrails 完整 | 偏 provider/runtime-first，不是 graph-first | 最小 Agent 原语、handoff、stream event、session、trace |
| LangGraph | graph/state/workflow-first | durable execution、checkpoint、HITL、state graph、time travel | 抽象偏底层，学习成本高 | 显式状态图、可恢复执行、图级编排 |
| Google ADK | 开放生态 + 多语言（含 Go）+ workflow agents | Go 支持、Sequential/Loop/Parallel、Session/State/Memory、MCP/A2A | 生态仍在快速演化 | Go 参考实现、workflow agent 分类、上下文管理 |
| Microsoft Agent Framework | Agent + Workflow 统一框架 | AutoGen + Semantic Kernel 合流，graph workflow、typed routing、middleware、telemetry | 生态新，长期形态还在收敛 | agents vs workflows 的边界、typed workflow、enterprise runtime |
| CrewAI | Crew（多 Agent 团队）+ Flow（流程骨架） | 角色任务模型直观，业务编排表达强 | 心智模型偏“业务工作流”，底层 runtime 可控性一般 | Flow 管状态、Crew 做智能协作的分层 |
| PydanticAI | 类型安全 agent framework | typed output、deps 注入、校验/重试、MCP/A2A、durable exec | Python-first，复杂图是后补能力 | 类型安全 tool/result contract、依赖注入、输出校验 |
| Mastra | TypeScript agent/workflow 平台 | agents、workflows、memory、MCP、eval、observability 一体化 | JS/TS 生态假设较重 | agent + workflow + memory + MCP 的产品化组合方式 |
| LlamaIndex Agents/Workflows | RAG-first agent/workflow | 数据索引/RAG 生态强、workflow event 语义清楚 | 通用 agent runtime 不是唯一中心 | RAG 工具化、数据上下文与 agent/workflow 的边界 |
| LangChain Agents | 工具调用 agent + middleware | model/tool middleware、structured output、多 Agent patterns、生态广 | 容易把框架层做厚 | middleware、tool/runtime context、subagent/handoff 模式 |
| Haystack Agents | Pipeline/RAG-first agent | pipeline/component 生态成熟，适合 agentic RAG | Agent 只是 Haystack 架构中的一个组件 | component pipeline + agentic RAG 的集成边界 |
| Dapr Agents | 基于 Dapr 的 durable agent/workflow | actor/workflow/runtime 基建、云原生分布式基础好 | 引入 Dapr 平台假设较重 | durable workflow、actor/state store 的基础设施抽象 |
| Agno | 全栈 agent/team framework | agents、teams、workflows、memory、knowledge、toolkit 覆盖广 | 产品表面很宽，需避免照搬成大而全 | team/workflow/product surface 的组织方式 |
| VoltAgent | TypeScript agent framework | agent、tool、workflow、observability、server/control plane 组合 | TS 生态绑定 | 开发者体验、可观测控制台、workflow 可视化思路 |
| BeeAI Framework | IBM 开源 agent framework | agent/tool/memory/workflow 与 provider 抽象 | 生态成熟度仍在演进 | provider-agnostic + workflow + memory 的组合 |
| smolagents | 轻量 CodeAgent | code-as-action、tool/managed agent 简洁 | Python/code-agent 专用倾向强 | coding agent harness、少抽象、可执行代码动作 |
| CAMEL | 多 Agent society / role-playing | 角色扮演、多 Agent 通信、数据生成/仿真 | 研究/仿真色彩强 | 角色协议、society 级协作模式 |
| MetaGPT | 软件公司式多 Agent | SOP、角色分工、软件工程流程表达强 | 更像应用框架，不适合做底层 runtime | SOP/work product/role pipeline 的产品化表达 |
| AutoGen（历史主流） | 多 Agent / group chat / event runtime | 多 Agent 影响力大，很多模式从它出圈 | **已进入 maintenance mode，微软推荐迁往 Microsoft Agent Framework** | 只适合借鉴历史模式，不建议作为未来主参考 |

### 3.2 各框架摘要

#### OpenAI Agents SDK

官方资料显示它强调**很少的 primitives**，核心就是：

- Agent
- tools
- handoffs / agents-as-tools
- guardrails
- sessions
- tracing
- human-in-the-loop
- MCP
- sandbox / resumable sessions

适合借鉴：

- **把 Agent runtime 原语做小**
- **把 handoff 做成一等能力**
- **把流式事件和 tracing 做成统一标准**

不适合直接照搬的点：

- 它更像“runtime + managed loop”，不是强 graph-first 编排
- 你的项目已经有 `workflow/` 层，不能把 workflow 能力全部塞回 agent runtime

#### LangGraph

LangGraph 明确把自己定位为：

- **low-level orchestration framework**
- 面向 **long-running, stateful agents**
- 核心强调：
  - persistence
  - durable execution
  - interrupts
  - time travel
  - memory
  - subgraphs

适合借鉴：

- **显式状态图**
- **checkpoint / resume / human interrupt**
- **把“agent loop”和“graph workflow”清楚分层**

这和本项目当前 `agent/` + `workflow/` 双层结构天然相容。

#### Google ADK

Google ADK 当前最值得关注的原因不是“Google 出品”，而是它在**架构形态上和本项目更接近**：

- 有 **Go SDK**
- agent 分成：
  - `LlmAgent`
  - workflow agents
    - `Sequential`
    - `Loop`
    - `Parallel`
  - custom agents
- 有：
  - Session / State / Memory
  - MCP
  - A2A
  - event loop
  - runtime / resume / evaluation / deployment

这对本项目最有参考价值，因为你本身就是 Go 项目，并且已经有：

- runtime builder
- streaming events
- MCP / A2A
- multi-agent / loop / parallel 模式

#### Microsoft Agent Framework

这是**当前微软主推的新一代 Agent 框架**。官方文档直接写明：

- 它是 **AutoGen + Semantic Kernel 的直接后继**
- 组合了：
  - AutoGen 的 agent abstraction
  - Semantic Kernel 的 session state / middleware / telemetry / type safety
  - graph-based workflows

我认为它最值得借鉴的不是具体 API，而是这条边界：

- **Agent**：开放任务、自治工具调用、计划/推理
- **Workflow**：有明确步骤、可控顺序、多个 agent/function 协同

这和本项目当前分层完全一致，也最适合作为架构判断准绳。

#### CrewAI

CrewAI 的表达方式很清楚：

- **Flow** = manager / process definition
- **Crew** = autonomous team

Flow 负责：

- state management
- event-driven execution
- control flow

Crew 负责：

- role-playing agents
- autonomous collaboration
- task delegation

这套分法对本项目也很有启发：

- `workflow/` 管“可控流程”
- `team/` 管“自治协作”
- 不要把两者混成一个巨型 multi-agent runtime

#### PydanticAI

PydanticAI 的优势在于**强类型契约**，官方文档明确强调：

- model-agnostic
- fully type-safe
- tool schema validation
- dependency injection
- MCP / A2A / UI event stream
- durable execution
- graph support

对本项目最值得借鉴的是：

- **工具参数与结果的强类型校验**
- **依赖注入（deps）**
- **类型安全的输出模型**
- **把 tool retry / validation error 变成框架级能力**

#### AutoGen（历史主流，但不建议作为未来主参考）

AutoGen 对业界影响很大，特别是：

- group chat
- event-driven runtime
- layered multi-agent design

但截至 **2026-04-23**，其 GitHub README 已明确标注：

- **Maintenance Mode**
- 新项目建议转向 **Microsoft Agent Framework**

因此对本项目的价值主要是：

- 用来理解历史多 Agent 模式来源
- 不适合作为未来 1~2 年的主对标对象

### 3.3 能力维度归纳：哪些能力应该融合到当前项目

本项目不应该把外部框架“整包搬进来”，而应该按能力维度融合。结合当前代码边界，建议如下：

| 能力维度 | 外部代表 | 当前项目已有基础 | 建议融合方式 | 优先级 |
|---|---|---|---|---|
| 最小 Agent 原语 | OpenAI Agents SDK、PydanticAI | `agent/runtime`、`sdk/runtime.go` | 收口到 `agent/runtime`：`Agent` / `Tool` / `Handoff` / `Session` / `RunConfig` / `RunEvent` | P0 |
| Agent 与 Workflow 分层 | Microsoft Agent Framework、CrewAI、Google ADK | `agent/` + `workflow/` 已分层但旧路径仍混杂 | `agent/team` 只表达自治协作，`workflow/runtime` 表达显式流程；禁止 `workflow` 反向依赖旧 `multiagent` | P0 |
| Durable execution / checkpoint | LangGraph、Dapr Agents、PydanticAI | `checkpoint_store`、executor tests | 把 checkpoint 从执行细节提升为 `RunEvent` / `WorkflowState` 的统一契约 | P1 |
| Typed tool/result contract | PydanticAI、Microsoft Agent Framework、LangChain | structured output / schema validator 已存在 | 统一 tool args/result schema、validation error、retry 语义；不要引入 Python 风格泛型 | P1 |
| Session / State / Memory | Google ADK、Mastra、Agno、BeeAI | `agent/capabilities/memory`、session 概念分散 | 明确 `Session` 是长期上下文，`Checkpoint` 是可恢复执行快照，`Memory` 是可检索长期知识 | P1 |
| Multi-agent team | CrewAI、AutoGen、CAMEL、MetaGPT、Agno | `agent/team`、`multiagent` | `agent/team` 作为唯一外部 facade；`multiagent` 降级为内部 engine/mode registry | P0 |
| RAG / knowledge integration | LlamaIndex、Haystack、Agno | `rag/`、tool/retrieval step | RAG 作为工具和上下文装配能力，不反向侵入 agent runtime | P2 |
| Middleware / guardrails | OpenAI Agents SDK、LangChain、Microsoft Agent Framework | guardrails、prompt defensive、authorization | 在统一 runtime 边界放置 middleware hooks：before model、before tool、after tool、after output | P2 |
| Observability / eval | OpenAI tracing、Mastra、VoltAgent、LangSmith | runtime stream events、evaluation 包 | 统一 trace id、run id、event schema；eval 不进入核心循环，只消费事件 | P2 |
| Coding agent harness | smolagents、Pi、OpenAI sandbox | hosted shell/file/code exec、MCP、subagent | 单独做 `CodingAgent Harness`，不要污染通用 `Agent` | P3 |

### 3.4 不建议融合的方向

- 不把所有框架的概念都变成公开 API；公开 API 只保留本项目的四个正式入口：`sdk.New(opts).Build(ctx)`、`agent/runtime`、`agent/team`、`workflow/runtime`。
- 不新增 `crew`、`graph`、`flow`、`society`、`pipeline` 等并行顶层入口；这些应映射到 `agent/team` 或 `workflow/runtime` 内部概念。
- 不为了兼容旧路径保留双轨 facade；开发阶段按仓库规则一次性收口旧实现。
- 不引入 Dapr / LangSmith / Mastra control plane 这类外部平台假设；只借鉴接口边界和事件模型。
- 不把 RAG-first 框架的索引/数据层直接塞进 Agent runtime；RAG 仍归 `rag/` 与工具/上下文装配层。

## 4. 对 AgentFlow 的参考实现建议

### 4.1 不建议“抄一个框架”，建议做组合参考

最适合本项目的路线，不是复制某一个框架，而是组合吸收：

### A. Runtime 内核：参考 OpenAI Agents SDK

保留极少的一等原语：

- Agent
- Tool
- Handoff
- Guardrail
- Session
- Trace / Stream Event

这可以帮助 `agent/runtime` 继续收口，而不是继续堆功能。

### B. 显式编排：参考 LangGraph / Microsoft Agent Framework

把以下能力坚定放在 `workflow/` 层，而不是继续塞进 `agent/`：

- graph orchestration
- checkpoint / resume
- human approval / HITL gate
- deterministic branching
- typed edge routing

### C. Go 实现路径：参考 Google ADK

尤其建议借鉴 ADK 的三类 workflow agents 思路：

- `Sequential`
- `Parallel`
- `Loop`

这和你当前 `multiagent/default_modes.go` 已经非常接近，可以顺势把它们升级为**正式、稳定、可复用的 workflow primitives**。

### D. 类型契约：参考 PydanticAI

建议加强这些点：

- tool args schema
- structured output schema
- dependency injection / runtime deps
- validation error -> retry / feedback
- eval-friendly typed result

### E. 团队协作 facade：参考 CrewAI

建议固定原则：

- `workflow` 负责状态与流程
- `team` 负责自治协作
- `multiagent` 保留为内部实现或 legacy surface

不要再让 `team`、`multiagent`、`workflow` 三者同时竞争“官方入口”。

### 4.2 建议的对外表面

我建议后续把正式公开表面明确收敛为。

> 下面的代码块是**建议中的 API 草图**，用于表达收口方向，不代表这些接口今天已经全部存在。

### 单 Agent

```go
rt, _ := sdk.New(opts).Build(ctx)
ag, _ := rt.NewAgent(ctx, cfg)
out, _ := ag.Execute(ctx, input)
```

### 多 Agent / Team

```go
t, _ := team.NewTeamBuilder("delivery").
    AddMember(researcher, "researcher").
    AddMember(reviewer, "reviewer").
    WithMode(team.ModeSupervisor).
    Build(logger)

res, _ := t.Execute(ctx, "分析需求并给出方案")
```

### 显式流程 / Workflow

```go
wf := workflowruntime.NewBuilder(...).
    WithCheckpointStore(store).
    WithHumanGate(hitl).
    Build()
```

设计原则：

- `sdk`：仓库级统一装配入口
- `agent/runtime`：单 Agent runtime
- `team`：官方多 Agent facade
- `workflow/runtime`：显式可控的 graph/workflow 编排

### 4.3 当前代码上最值得优先做的 4 件事

### P0：先修文档一致性

优先级最高。

至少要清理这些旧口径：

- `github.com/BaSui01/agentflow/agent` root import
- `agent/base.go` / `agent/react.go` / `agent/completion.go`
- `agent/teamadapter`

如果不做这一步，外部开发者会被旧文档带偏。

### P1：把 `team` 和 `multiagent` 的边界彻底收口

目标：

- **`team` 成为唯一官方多 Agent 表面**
- `multiagent` 只保留内部模式注册与执行器，不再直接暴露为主推荐 API

当前 `internal/usecase/agent_service.go:227` 仍直接调用 `multiagent.GlobalModeRegistry().Execute(...)`，这会稀释官方入口。

### P2：把 `loop/parallel/sequential` 从 legacy mode 提升为正式 workflow primitive

建议把这些能力从“字符串模式”升级为：

- 可组合节点
- 可 checkpoint
- 可 tracing
- 可带 typed state

### P3：给 runtime 补更强的 typed contract

优先方向：

- structured output
- tool arg/result validation
- event schema
- per-run dependency injection

## 5. 总判断

### 5.1 对当前项目的判断

我给当前 Agent 框架的判断是：

- **内部真实可用性：高**
- **架构方向：对**
- **对外产品化成熟度：中**

一句话总结：

> **它已经是一个能跑的 Agent 框架内核，但还不是一个“文档彻底收口、对外表面极简、易于传播”的成品框架。**

### 5.2 对参考实现方向的判断

如果后续要继续演进，我建议以这条组合路线为主：

1. **OpenAI Agents SDK** 学最小原语和 runtime loop
2. **LangGraph / Microsoft Agent Framework** 学 graph + checkpoint + HITL
3. **Google ADK** 学 Go 侧 workflow agent 分类、session/state/memory、MCP/A2A
4. **PydanticAI** 学 typed contracts / deps / validation
5. **CrewAI** 学 Flow 与 Team 的职责切分
