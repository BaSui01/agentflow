# To PRD — 生成 PRD

基于当前对话上下文和代码库理解产生 PRD。**不要访谈用户**——仅综合已有信息。

## 流程

1. 探索代码库以了解当前状态（如果还未探索）。整个 PRD 使用项目领域词汇表，并尊重相关 ADR

2. 画出需要构建或修改的主要模块。主动寻找可提取为**深模块**的机会——能独立测试的、封装了大量功能的、接口简单的模块。
   与用户确认模块划分是否符合预期

3. 使用以下模板编写 PRD，然后发布到 Issue 追踪器

## PRD 模板

### 问题陈述

用户面临的问题，从用户视角描述。

### 解决方案

解决方案，从用户视角描述。

### 用户故事

按编号列表。每个故事格式：

> 作为一个 <角色>，我想要 <功能>，以便 <收益>

列表应尽可能详尽，覆盖功能的所有方面。

### 实现决策

包括：
- 要构建/修改的模块
- 模块的接口
- 技术澄清
- 架构决策
- Schema 变更
- API 契约

**不要**包含具体文件路径或代码片段，它们会很快过时。

### 测试决策

- 好测试的标准（只测外部行为，不测实现细节）
- 哪些模块需要测试
- 测试的先例（代码库中类似类型的测试）

### 超出范围

明确说明本次不包含的内容。

### 补充说明

任何其他相关信息。
