# Prototype — 原型开发

原型是**回答某个问题的实验性代码**。问题决定了形态。

## 选择路径

根据要回答的问题选择路径：

- **"这个逻辑/状态模型感觉对吗？"** → 构建小型的交互式终端应用，推动状态机经历难以在纸上推理的案例
- **"这应该长什么样？"** → 在单一路由上生成多个截然不同的 UI 变体，通过 URL 参数切换

两条路径产出截然不同的制品。如果问题确实模糊且用户不可达，默认根据代码上下文选择（后端模块 → 逻辑；页面/组件 → UI），并在原型顶部说明假设。

## 两条路径通用规则

1. **从第一天起就是一次性代码，并明确标记。** 放在将被使用的位置附近，但命名上能看出它不是生产代码
2. **一个命令就能运行。** 用户必须能无脑启动
3. **默认无持久化。** 状态在内存中。持久化是原型正在**验证**的东西，不是依赖
4. **跳过打磨。** 无测试、无错误处理（除非让原型_能运行_）、无抽象。目的是快速学习然后删除
5. **暴露状态。** 每次操作后打印或渲染完整相关状态，让用户看到变化
6. **完成时删除或吸收。** 答案验证后，要么删除原型，要么将决策融入真实代码——不要让它腐烂在仓库中

## 完成后

**答案**是原型唯一值得保留的东西。捕获到持久化的地方（commit message、ADR、Issue 或原型旁的 NOTES.md），附上它回答的问题。如果用户在场，快速对话即可捕获；如果不在，留下占位符供下次填充裁决结果。
