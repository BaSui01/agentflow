# To Issues — 拆解为 Issue

将计划拆解为独立的可执行 Issue，每个 Issue 是一个**垂直切片**。

## 流程

### 1. 收集上下文

从对话上下文中获取已有的信息。如果用户传入了 Issue 引用，从 Issue 追踪器获取完整内容。

### 2. 探索代码库（可选）

如果还未探索代码库，先了解当前代码状态。Issue 标题和描述使用项目领域词汇表，并尊重相关 ADR。

### 3. 起草垂直切片

将计划拆解为 **tracer bullet** Issue。每个 Issue 是一个薄的垂直切片，贯穿**所有集成层**，不是某一层的水平切片。

切片分为两种：
- **HITL** — 需要人工介入（架构决策、设计评审等）
- **AFK** — 可全自动实现和合并

规则：
- 每个切片提供一条窄但**完整**的路径贯穿所有层（schema、API、UI、测试）
- 完成的切片可独立演示或验证
- 宁多薄切片，勿少厚切片

### 4. 与用户确认

展示拆解方案。每条展示：
- **标题**：简短描述
- **类型**：HITL / AFK
- **被阻塞于**：哪些前置切片
- **覆盖的用户故事**：如果源材料有

询问用户粒度是否合适、依赖关系是否正确、分组是否合理。

### 5. 发布 Issue

按依赖顺序发布（阻塞者先发，以便引用真实 Issue ID）。

Issue 模板：
```
## 父级
（如有来源 Issue）

## 构建内容
简洁描述端到端行为，而非逐层实现。避免具体文件路径。

## 验收标准
- [ ] 标准 1
- [ ] 标准 2

## 被阻塞于
- 阻塞 Issue 引用（或无）
```
