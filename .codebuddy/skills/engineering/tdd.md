# TDD — 测试驱动开发

## 哲学

测试应测试**行为**而非**实现**。

| WRONG | RIGHT |
|-------|-------|
| 测试私有方法 | 通过公共接口测试行为 |
| 使用 Mock 验证方法调用 | 使用 Fake 验证状态和输出 |
| 测试与实现紧耦合 | 测试解耦于实现 |
| 测试在文件内部 | 测试在文件外部 |
| 测试间共享可变状态 | 每个测试隔离运行 |

## 反模式：水平切片

不要一次性写完所有测试再写实现。而是用**垂直增量**：一个完整的端到端切片 → 测试通过 → 下一步。

```
WRONG:
  1. 写所有模型的测试
  2. 写所有视图的测试
  3. 写所有 API 的测试
  4. 写所有实现

RIGHT:
  1. 写一个 tracer bullet 测试（贯穿所有层）
  2. 让它通过
  3. 写下一个 tracer bullet 测试
  4. 让它通过
  ...
```

## 工作流

### 1. 计划

确定下一个垂直切片，理解它跨越的层。

### 2. Tracer Bullet

写一个贯穿所有集成层的端到端测试。场景必须是完整、有意义的用户旅程。测试仅通过公共接口与系统交互。

- [ ] 测试只测试公开行为，不依赖实现细节
- [ ] 测试仅使用公共 API/函数签名
- [ ] 测试能抵御重构（改变内部实现不破坏测试）

### 3. 增量循环

在 tracer bullet 之后，逐步增补更多测试，一次一个。保持测试快速。

每个循环结束后检查：
- [ ] 测试只测行为，不是实现
- [ ] 测试仅通过公共接口
- [ ] 测试能抵御重构
- [ ] 测试命名清晰表达场景和预期

### 4. 重构

当测试全部通过后，可以安全地重构。重构期间测试保持绿色。

- [ ] 所有测试绿色
- [ ] 没有重复代码
- [ ] 命名表达意图
- [ ] 模块边界合理
