# Improve Codebase Architecture — 改进架构

识别架构摩擦点，提出**深化机会**——将浅模块重构为深模块。目标是可测试性和 AI 可导航性。

## 核心术语

- **模块** — 任何有接口和实现的东西（函数、类、包）
- **接口** — 调用者使用模块必须知道的一切：类型、不变量、错误模式、顺序、配置
- **实现** — 内部代码
- **深度** — 接口的杠杆率：小接口承载大行为。**深 = 高杠杆**，**浅** = 接口几乎和实现一样复杂
- **接缝** — 接口所在位置；可在不就地修改的情况下改变行为的地方
- **适配器** — 在接缝处满足接口的具体实现
- **杠杆** — 调用者从深度中获得的价值
- **局部性** — 维护者从深度中获得的价值：变更、bug、知识集中在一处

关键原则：
- **删除测试**：想象删除这个模块。如果复杂性消失，说明它是透传。如果复杂性分散到 N 个调用者，说明它有价值
- **接口即测试面**
- **一个适配器 = 假设接缝，两个适配器 = 真实接缝**

## 流程

### 1. 探索

先阅读领域词汇表和相关 ADR。然后探索代码库，记录遇到的摩擦：

- 理解一个概念需要跳跃多少个模块？
- 哪些模块是**浅的**——接口几乎和实现一样复杂？
- 哪些纯函数只是为了可测试性提取的，但真正的 bug 隐藏在调用方式中？
- 哪些紧耦合模块跨越了接缝？
- 哪些代码未测试，或者通过当前接口难以测试？

对怀疑是浅模块的做**删除测试**：删除它会集中复杂性还是只是转移它？

### 2. 展示候选

按编号列出深化机会。每条包括：

- **文件** — 涉及的模块/文件
- **问题** — 当前架构为什么造成摩擦
- **方案** — 白话描述会改变什么
- **收益** — 从局部性和杠杆角度说明，以及测试如何改进

使用 CONTEXT.md 的领域语言描述领域，使用核心术语描述架构。
**ADR 冲突**：仅当摩擦真实到值得重新审视 ADR 时才提出冲突。

不要提出接口方案。先问用户："你想探索哪一个？"

### 3. 访谈循环

用户选定后进入深度访谈。决策结晶时的副作用：

- 命名了一个不在 CONTEXT.md 中的概念？添加到 CONTEXT.md
- 澄清了模糊术语？立即更新 CONTEXT.md
- 用户因有分量的理由拒绝了候选？提供 ADR："需要我记录为 ADR 以免未来再次建议吗？"
- 想探索深化模块的替代接口？进行接口设计讨论
