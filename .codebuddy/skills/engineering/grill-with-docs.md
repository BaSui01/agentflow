# Grill With Docs — 深度访谈 + 文档化

对计划的每个方面进行深度访谈，直到达成共同理解。逐条走通设计树的分支，逐个解决决策间的依赖。每次只问一个问题，等待反馈后继续。

如果问题可以通过探索代码库回答，先探索代码库。

## 领域意识

探索代码库时同时查找现有文档：

### 单上下文结构（最常见）
```
/
├── CONTEXT.md
├── docs/adr/
│   ├── 0001-event-sourced-orders.md
│   └── 0002-postgres-for-write-model.md
└── src/
```

### 多上下文结构（有 CONTEXT-MAP.md 时）
```
/
├── CONTEXT-MAP.md
├── docs/adr/
├── src/ordering/
│   ├── CONTEXT.md
│   └── docs/adr/
└── src/billing/
    ├── CONTEXT.md
    └── docs/adr/
```

文件延迟创建 — 只有当有内容要写时才创建。如果不存在就创建。

## 访谈过程中的行动

### 对照术语表挑战

当用户使用的术语与 `CONTEXT.md` 中现有语言冲突时，立即指出。

### 精炼模糊语言

当用户使用模糊或过载的术语时，提出精确的规范术语。

### 讨论具体场景

用具体场景进行压力测试，探测边界情况，迫使用户在概念边界上精确化。

### 交叉引用代码

当用户描述系统行为时，检查代码是否一致。发现矛盾立即指出。

### 内联更新 CONTEXT.md

当术语达成一致后，立即更新 `CONTEXT.md`，不要批量处理。只包含对领域专家有意义的术语，不耦合实现细节。

### 审慎提供 ADR

仅当以下三个条件**全部满足**时才创建 ADR：
1. **难以逆转** — 改变主意成本高
2. **没有上下文则令人惊讶** — 未来读者会疑惑"为什么这么做？"
3. **真实权衡的结果** — 存在真正的替代方案且有选择理由

缺少任意一条则跳过 ADR。
