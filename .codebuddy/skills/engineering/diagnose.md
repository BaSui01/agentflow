# Diagnose — Bug 诊断

按纪律执行六阶段调试循环。**没有可靠的反馈循环就不要跳到假设阶段。**

## 阶段 1 — 构建反馈循环（核心技能）

目标是获得一个**快速、确定、可重复执行的 pass/fail 信号**。

优先级顺序：
1. 写一个能触发 bug 的失败测试（unit / integration / e2e）
2. 用 curl / HTTP 脚本调用 dev server
3. CLI 调用 + 对比 fixture 输出快照
4. 无头浏览器脚本（Playwright）
5. 重放捕获的 trace（网络请求、事件日志等）
6. 临时最小化 harness（只保留触发 bug 的代码路径）
7. 属性/模糊测试循环（随机输入 1000 次）
8. 二分回测 harness（`git bisect run`）
9. 差异循环（新旧版本或不同配置下对比输出）
10. 人辅助脚本（万不得已时）

**迭代优化循环**：加快速度、让信号更尖锐、更确定（固定种子、隔离 IO 等）。

对非确定性 bug：目标是提高复现率，而非完全确定；从 1% 提到 50% 即可调试。

**若实在无法构建循环**：必须明确告知用户，列出尝试过的方法，索要环境或生产埋点权限。

## 阶段 2 — 重现

- [ ] bug 症状与用户描述一致
- [ ] 多次运行可复现（或非确定 bug 有足够高复现率）
- [ ] 已记录确切症状（错误信息、错误输出、慢的时间数值）

## 阶段 3 — 假设

同时提出 **3–5 个可证伪的假设**，每个必须附带预测：

> 如果 <X> 是原因，那么 <改变 Y> 会使 bug 消失 / <改变 Z> 会使 bug 更糟

**在测试之前将假设列表展示给用户**（用户可能知道已有排除项或最近部署变更）。

## 阶段 4 — 插桩测试

每个探测必须对应一个具体假设预测，**一次只改变一个变量**。

- 工具优先级：调试器/REPL > 有目标日志 > 绝不"全量日志再 grep"
- **日志必须带唯一 tag**（如 `[DEBUG-a4f2]`），方便后期一次性清理
- 性能回归：先建立基线测量，再二分，不要依赖日志估算时间

## 阶段 5 — 修复 + 回归测试

1. 在正确的 seam 上写**失败的测试** → 观察它失败
2. 应用修复 → 观察它通过
3. 再次用原始场景（未最小化的）运行阶段 1 循环

如果不存在正确的 seam（架构问题导致无法锁定 bug），记录该发现，后续交给 improve-codebase-architecture。

## 阶段 6 — 清理 + 事后复盘

- [ ] 原始 bug 不再复现（重跑阶段 1 循环）
- [ ] 回归测试通过（或已记录无 seam）
- [ ] 所有 `[DEBUG-...]` 日志已移除
- [ ] 临时原型已删除或移动到明确标记的调试目录
- [ ] 被验证的假设记录在 commit / PR message 中

**思考：什么能预防这个 bug？** 如果答案是架构改进，则提交 issue 给 improve-codebase-architecture 技能。**修复之后再做，而不是之前**——此时你掌握的信息最多。
